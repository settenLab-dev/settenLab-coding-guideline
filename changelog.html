Changelog

Purpose

This page documents releases of the Coding Guideline itself: what changed, why it changed, and what teams should do (if anything) as a result. Use it as the source of truth when the guideline evolves.

Current version

v1.0.0
Status: stable baseline

Audience

Engineers, reviewers, and maintainers who need to understand guideline changes and their impact on existing projects.

Versioning policy

The guideline follows Semantic Versioning: MAJOR.MINOR.PATCH.

MAJOR

Breaking changes to rules or structure that can require project updates or rework. Example: renaming required CSS conventions, changing baseline folder standards, or replacing default component patterns.

MINOR

Backwards-compatible additions. Example: new section, new recommended patterns, additional examples, or expanded checklists that do not invalidate existing projects.

PATCH

Backwards-compatible fixes and clarifications. Example: typos, wording fixes, corrected code samples, improved explanations, minor navigation improvements.

Pre-release labels

  • v1.2.0-beta.1 ‚Äî preview for feedback
  • v1.2.0-rc.1 ‚Äî release candidate

Use pre-releases when the change set is substantial or needs validation across multiple projects.

How versions are managed

Keep versioning predictable and auditable. The guideline should be updated through pull requests with explicit change notes.

Single source of version

Store the current version in one place (recommended: CHANGELOG.md or a small config file) and mirror it in the UI footer (.sidebar__version) as part of the release PR.

Release PR requirements

  • Bump version number
  • Add changelog entry (date, summary, impact)
  • Link to PRs/issues (if applicable)
  • Confirm navigation and anchors still work

Git tags

Tag releases with the version number (e.g. v1.1.0) to enable quick diffing between releases.

  • Changed ‚Äî updated recommendations
  • Deprecated ‚Äî planned removal, migration notes
  • Removed ‚Äî deleted rules/patterns
  • Fixed ‚Äî corrections

Governance

The guideline is a product. Changes should be reviewed with the same rigor as code, especially when they introduce team-wide obligations.

What is a breaking change

  • Rule becomes ‚Äúmust‚Äù where it used to be optional
  • Renaming conventions that invalidate existing code
  • New required tooling or CI checks
  • Removing a previously recommended approach

Deprecations

Deprecate before removing. A deprecation must include: rationale, migration guidance, and a planned removal version.

Editorial changes

Spelling fixes and clarification-only edits are patch releases, unless they effectively change interpretation of a rule.
