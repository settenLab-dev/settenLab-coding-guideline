Typography & Font Loading

Overview

Typography is a core part of our visual identity. This page defines how we choose font stacks, structure font files, and load them efficiently in the browser. The goal is to keep text readable, predictable, and performant across devices.

Goals

  • Use a small, consistent font family set across projects.
  • Load fonts without blocking rendering or causing CLS.
  • Prefer local .woff2 over external CDNs.
  • Base typography on rem with a predictable scale.

Font stack, Japanese typography & line breaking

We use Google Fonts widely across projects, but Japanese text requires additional rules. This section defines a token-based font system, how to place YakuHan in the stack, and how to handle safe line breaks.

Project rules

  • Use tokens. Do not hardcode font-family in components.
  • Japanese punctuation: keep "YakuHanJP" / "YakuHanMP" first in JP stacks.
  • Apply JP behavior via :lang(ja) or per-block lang="ja".
  • Use <wbr> only when you intentionally control breaking points (URLs, long IDs, mixed JP/EN UI strings).

Single source of truth: typography tokens

:root {
  --sans-font: "YakuHanJP", "Public Sans", sans-serif;
  --serif-font: "YakuHanMP", "Shippori Mincho B1", serif;
  --gothic-font: "YakuHanJP", "Zen Kaku Gothic Antique", sans-serif;
  --garamond: "EB Garamond", serif;
  --sans-serif-mix: "YakuHanJP", "Public Sans", "Zen Kaku Gothic Antique", sans-serif;
}

Tokens allow projects to switch font families without rewriting components.

Default usage (recommended)

body {
  font-family: var(--sans-font);
}

.type-gothic { font-family: var(--gothic-font); }
.type-serif  { font-family: var(--serif-font); }
.type-mix    { font-family: var(--sans-serif-mix); }

Language scoping

/* Apply JP behavior only where the content is Japanese */
:lang(ja) {
  font-family: var(--sans-font);
}

Best practice: set lang="ja" at page level for JP pages, or per section for mixed-language pages.

Google Fonts (common in projects)

Google Fonts are allowed. For production, prefer self-hosting when possible, but projects may keep Google Fonts for speed of delivery and operational simplicity.

Example: preconnect + stylesheet

<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<link
  href="https://fonts.googleapis.com/css2?
  family=Public+Sans:wght@400;600;700&
  family=Zen+Kaku+Gothic+Antique:wght@400;700&
  family=Shippori+Mincho+B1:wght@400;700&
  display=swap"
  rel="stylesheet"
/>

Keep weights minimal (typically 400 + 700). Avoid loading unused styles.

YakuHan is a punctuation-only font. It must be placed first in the Japanese stack. Use YakuHanJP for Gothic/sans, and YakuHanMP for Mincho/serif.

--gothic-font:"YakuHanJP", "Zen Kaku Gothic Antique", sans-serif;
--serif-font: "YakuHanMP", "Shippori Mincho B1", serif;

If you use a different JP family, keep YakuHan first and replace only the base fonts.

Controlled line breaks with <wbr>

Japanese text does not use spaces, and mixed JP/EN strings often overflow. Insert <wbr> only where breaking is acceptable (URLs, long tokens, product codes, "JP + English" UI).

HTML examples

<p class="u-wbr">
  https://example.com/super-long-path<wbr>/with<wbr>/segments
</p>

<!-- Mixed UI (JP + EN) -->
<p lang="ja" class="type-mix u-wbr">
  „Éó„É©„É≥: Enterprise<wbr>Support<wbr>Package

CSS rule (apply only when you use wbr)

/* Preferred: auto-detect elements that contain direct child <wbr> */
.u-wbr:has(> wbr) {
  word-break: keep-all;
  overflow-wrap: anywhere;
}

Note: :has() is not universal in older browsers. Use the fallback class approach below if you need it.

Important note about rem base

This project uses html { font-size: 62.5%; } which makes 1rem = 10px. Keep typography values in rem and avoid mixing with px. Treat the root size as a project-level decision.

Font formats & folder structure

Projects must use only modern, compressed formats for custom fonts. Legacy formats are not allowed in new projects.

Allowed formats

  • woff2 ‚Äì required, default for all custom fonts.

Not allowed

  • .ttf, .otf ‚Äì large and inefficient.
  • .eot, .svg, .woff ‚Äì legacy formats.

Folder structure

assets/fonts/
  inter-regular.woff2
  inter-semibold.woff2
  jetbrains-mono-regular.woff2

Font files are named in English with lowercase and kebab-case, including family and weight.

Font loading strategy

Fonts must be loaded in a way that does not block rendering and does not cause layout shifts when they appear.

Recommended pattern

We load only the required weights as .woff2 with font-display: swap. 

<!-- In <head>: preload critical fonts -->
<link
  rel="preload"
  href="assets/fonts/inter-regular.woff2"
  as="font"
  type="font/woff2"
  crossorigin
/>

<link
  rel="preload"
  href="assets/fonts/inter-semibold.woff2"
  as="font"
  type="font/woff2"
  crossorigin
/>
/* In CSS: define @font-face */
@font-face {
  font-family: "Inter";
  src: url("../fonts/inter-regular.woff2") format("woff2");
  font-weight: 400;
  font-style: normal;
  font-display: swap;
}

@font-face {
  font-family: "Inter";
  src: url("../fonts/inter-semibold.woff2") format("woff2");
  font-weight: 600;
  font-style: normal;
  font-display: swap;
}

Local fonts vs CDN

For production projects, we prefer local font hosting. CDNs (e.g. Google Fonts) are allowed only for prototypes or non-critical marketing pages.

Local fonts (recommended)

Full control, no external dependencies, predictable caching and performance.

  • Only .woff2 files in assets/fonts/.
  • Preload critical weights (regular, semibold).
  • Use font-display: swap.

CDN fonts (optional)

May be used for prototypes or temporary pages, but can add latency and complexity.

  • Requires preconnect to host domain.
  • Less control over caching and availability.
  • Only for non-critical projects.

CDN example (for prototypes only)

<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link
  href="https://fonts.googleapis.com/css2?family=Inter:wght@400;600&display=swap"
  rel="stylesheet"
/>

Performance & CLS

Font loading directly affects performance metrics such as Largest Contentful Paint (LCP) and Cumulative Layout Shift (CLS).

Rules

  • Use minimum number of weights per family (usually 400 and 600).
  • Always set font-display: swap (or optional for highly performance-sensitive pages).
  • Do not load italics or extra weights unless they are required by design.
  • Use system fonts for small UI elements (labels, helper text) when appropriate.

font-display options

/* Default and recommended */
font-display: swap;      /* text appears immediately with fallback */

/* For extreme performance-sensitive pages */
font-display: optional;  /* do not block, may never swap if too slow */

Font sizes & typographic scale

Font sizes must follow design first. These recommendations define a safe baseline scale, accessibility minimums, and responsive rules that work well for Japanese and mixed JP/EN UI.

Rules

  • Design is the source of truth. Use the scale below only when design is missing or for internal docs/components.
  • This project uses html { font-size: 62.5%; }, so 1rem = 10px. Keep all sizes in rem.
  • Prefer limited set of sizes (tokens) over arbitrary values. Avoid "creative" one-off sizes.
  • For Japanese UI, prefer slightly larger minimums and comfortable line-height (dense kanji becomes hard to read quickly).

Recommended baseline scale (1rem = 10px)

/* Text sizes */
--fs-12: 1.2rem; /* 12px (legal notes only) */
--fs-14: 1.4rem; /* 14px (secondary, captions) */
--fs-16: 1.6rem; /* 16px (default body) */
--fs-18: 1.8rem; /* 18px (comfortable reading / JP-heavy pages) */

/* Headings */
--fs-20: 2.0rem; /* 20px */
--fs-24: 2.4rem; /* 24px */
--fs-28: 2.8rem; /* 28px */
--fs-32: 3.2rem; /* 32px */
  • Body default: 1.6rem (16px).
  • Small text minimum: 1.4rem (14px) for UI labels/captions.
  • 12px (1.2rem): only for legal/disclaimer, never for core UI.

Recommended line-height

/* Body */
--lh-body: 1.6;   /* best default */

/* Dense UI labels/buttons */
--lh-ui: 1.2;

/* Headings */
--lh-heading: 1.2; /* 1.15‚Äì1.25 depending on font */

For Japanese paragraphs, line-height below 1.5 often feels cramped.

Responsive sizing (preferred pattern)

Use clamp() for headings to avoid "jumps" and keep proportions stable. Because you use container queries, you may also use cqw. 

/* Example: responsive H1 within a container */
.type-h1 {
  font-size: clamp(2.6rem, 2.2rem + 1.2cqw, 3.6rem);
  line-height: var(--lh-heading);
}

/* Example: responsive H2 */
.type-h2 {
  font-size: clamp(2.2rem, 2.0rem + 0.8cqw, 3.0rem);
  line-height: var(--lh-heading);
}

Keep body text typically fixed (1.6rem) and scale headings instead.

Usage recommendations

  • Use gothic/sans for headings in JP UI by default (better clarity).
  • Use serif only for editorial/article blocks (not buttons/forms).

Typography Sizes & Design Source of Truth

Typography sizes are always defined by the design. The guideline provides a unified method of implementation (units, scaling, responsiveness), but it does not enforce fixed font-size values across projects. Each product, page, or component must use the exact sizes specified in the design source.

Key Principles

  • Design is the single source of truth. All font sizes, weights, letter spacing, and line heights must match the provided design.
  • Developers implement the typography system, not the size values. Use rem, consistent scaling, and mobile-first responsive rules.
  • Fallback sizes exist only for development without ready design. They must be replaced before release.
  • Do not modify root font-size. html { font-size: 16px; } remains default to avoid accessibility issues.
  • Do not invent new text styles in code. Only use what's defined in the design system.

Why sizes depend on the design

  • Different projects require different hierarchy (promo vs dashboard).
  • Visual rhythm and spacing come from design, not from code templates.
  • Text contrast, weight distribution, and balance vary depending on branding.
  • Developers should not "guess" typographic scaling ‚Äî only implement it.

Fallback scale (use only when design is missing)

These values are temporary and must be replaced with designer-provided sizes once they are available. 

html {
  font-size: 16px;
}

body {
  font-size: 1rem;      /* 16px */
}

h1 { font-size: 2rem; }     /* 32px */
h2 { font-size: 1.75rem; }  /* 28px */
h3 { font-size: 1.5rem; }   /* 24px */

.small-text {
  font-size: 0.875rem;  /* 14px */
}

/* Optional: desktop enhancements */
@media (min-width: 768px) {
  h1 { font-size: 2.25rem; }
  h2 { font-size: 2rem; }
}

Font loading patterns

Comparison of common font loading patterns and our recommended approach for new projects.

Bad: no preload, external only

Render-blocking CSS from Google Fonts, no control.

The page waits for external CSS. On slow networks, text may be invisible or shift when fonts load.

Good: local woff2 + preload

Local fonts, preloaded, swap behavior.

Fonts are loaded from assets/fonts/, preloaded for first paint, and do not block rendering.

Advanced: partial loading

Only critical weights are preloaded.

On performance-critical pages, load only base weights on first view, and lazy-load others if needed.

<link
  rel="preload"
  href="assets/fonts/inter-regular.woff2"
  as="font"
  type="font/woff2"
  crossorigin
