Overview

This page defines how we author and organize CSS/SCSS for predictable, scalable UI. It aligns with our project file-structure rules and uses BEM as the primary naming convention. SCSS sources compile to assets/css/style.css (recommended: Live Sass Compiler by Glenn Marks with workspace settings).

Core Principles

  • Design tokens first: use CSS custom properties for colors, spacing, typography
  • Component isolation: styles are authored per component and composed via BEM
  • Low specificity: prefer single-class selectors; avoid deep nesting
  • Progressive enhancement: the base experience must work without advanced CSS
  • Consistency: one architecture, one naming system, one formatting standard

Styling File Structure

Keep styling assets aligned with the project structure. Use kebab-case for file and folder names. For component-based projects, co-locate component HTML/CSS/JS together (proximity principle).

Live Sass Compiler (Glenn Marks) ‚Äî recommended workspace setup

Use workspace settings to make SCSS compilation consistent across the team. Compile SCSS into assets/css/ and link that CSS in HTML.

In VS Code, start compilation via the status bar ‚ÄúWatch Sass‚Äù button (or use the command palette for one-time compilation).

  • Keep sources in assets/scss/ and treat assets/css/ as generated output (do not edit by hand).
  • Prefer expanded CSS + sourcemap for development; optionally also emit a compressed .min.css for production.
  • If you need vendor prefixes, the extension can apply Autoprefixer rules via a Browserslist-style query. 
// .vscode/settings.json (workspace)
{
  "liveServer.settings.port": 5501,

  "liveSassCompile.settings.formats": [
    {
      "format": "expanded",
      "extensionName": ".css",
      "savePath": "./assets/css",
      "generateMap": true
    },
    {
      "format": "compressed",
      "extensionName": ".min.css",
      "savePath": "./assets/css",
      "generateMap": false
    }
  ],
  "liveSassCompile.settings.excludeList": [
    "**/node_modules/**",
    "**/.vscode/**",
    "**/.history/**"
  ],
  "liveSassCompile.settings.autoprefix": [
    "> 1%",
    "last 2 versions"

}

CSS/SCSS Architecture

We use an opinionated layering model: tokens ‚Üí reset ‚Üí base ‚Üí layout ‚Üí components ‚Üí utilities ‚Üí overrides. This keeps specificity under control and makes overrides intentional.

Reset / Baseline (Recommendation)

Prefer a modern normalization (e.g., modern-normalize) plus a small, opinionated baseline layer. Avoid ‚Äúhard resets‚Äù that erase semantic defaults (lists, headings) and can harm accessibility.

  • Load first: base/normalize.css (or modern-normalize) ‚Üí base/baseline.css ‚Üí tokens ‚Üí components ‚Üí utilities ‚Üí overrides
  • Keep it minimal: box-sizing, body margin, media defaults, form font inheritance, and focus-visible styling
  • Do not style via JS hooks or IDs; resets should be framework-agnostic and stable 
/* base/baseline.css (example) */
*, *::before, *::after { box-sizing: border-box; }

html { -webkit-text-size-adjust: 100%; }
body { margin: 0; }

img, video, canvas, svg { display: block; max-width: 100%; }

input, button, textarea, select { font: inherit; }

/* Ensure a visible focus indicator (use a token/variable) */
:focus-visible {
  outline: 2px solid var(--focus-ring, currentColor);
  outline-offset: 2px;
}

/* Respect reduced motion */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after { scroll-behavior: auto; transition: none !important; animation: none !important; }
}

If you use CSS layers, place reset/normalize + baseline in @layer base. If you do not use a dedicated normalize file, document the decision and keep baseline.css explicit.

Reset / Baseline: implementation notes

Practical setup in SCSS (_reset.scss + _base.scss)

Keep normalization and base styles explicit. Recommended split:

  • _reset.scss: normalization/consistent defaults (box-sizing, media defaults, form font inheritance)
  • _base.scss: project baseline (typography defaults, focus-visible, reduced-motion, link styles)
  • Tokens stay separate (_tokens.scss). Components never depend on reset internals. 
/* style.scss (entrypoint) */
@use "reset";
@use "tokens";
@use "base";
@use "components";
@use "utilities";
@use "overrides";
How to combine with @layer (no runtime CSS @import)

If you use cascade layers, each partial defines its own layer. This avoids CSS @import and keeps ordering predictable. 

/* _reset.scss */
@layer reset {
  *, *::before, *::after { box-sizing: border-box; }
  body { margin: 0; }
  img, video, canvas, svg { display: block; max-width: 100%; }
  input, button, textarea, select { font: inherit; }
}

/* _base.scss */
@layer base {
  :focus-visible {
    outline: 2px solid var(--focus-ring, currentColor);
    outline-offset: 2px;
  }

  @media (prefers-reduced-motion: reduce) {
    *, *::before, *::after { transition: none !important; animation: none !important; }
  }
}

Recommended layer order

/* style.scss (entrypoint) */
@layer tokens, reset, base, layout, components, utilities, overrides;

/* Load modules. Each module owns its @layer block internally (recommended). */
@use "tokens";
@use "reset";
@use "base";
@use "layout";
@use "components";
@use "utilities";
@use "overrides";

Prefer Sass modules (@use / @forward). Avoid legacy global @import; place each file‚Äôs rules inside its own @layer block to keep cascade order explicit.

Mixins (Sass/SCSS)

Use mixins for cross-cutting concerns (media queries, a11y helpers, platform quirks). Avoid using mixins for design tokens (colors/spacing) or to generate component selectors.

  • Good use: focus ring + forced-colors fallback, reduced-motion wrappers, repeated media/query patterns
  • Avoid: mixins that set hard-coded colors/spacing (use CSS variables + tokens instead)
  • One responsibility per mixin; keep side effects minimal and document each mixin with a short example
  • Prefer @use/@forward modules; avoid legacy global @import in new code 
// _mixins.scss (example)
@mixin motion-safe {
  @media (prefers-reduced-motion: no-preference) {
    @content;
  }
}

@mixin focus-ring {
  &:focus-visible {
    outline: 2px solid var(--focus-ring, currentColor);
    outline-offset: 2px;
  }

  @media (forced-colors: active) {
    &:focus-visible { outline: 2px solid ButtonText; }
  }
}

// usage
.button {
  @include focus-ring;
  @include motion-safe { transition: transform 120ms ease; }
}

Layout & Containers

Choose a single approach to section structure and containers for the entire project. Do not mix different approaches within one project.

Rules for All Approaches

  • One approach per project: choose one method and stick to it everywhere
  • Adaptive gutters: use clamp() or min()/max() for gutters
  • Consistency: identical spacing between sections across all pages
  • Minimalism: avoid extra wrappers and deep nesting

Layout Tokens (Shared Across All Approaches)

/* _tokens.scss */
:root {
  /* Base gutters */
  --gutter: clamp(16px, 4vw, 40px);
  --gutter-sm: clamp(12px, 2vw, 24px);
  --gutter-lg: clamp(24px, 6vw, 64px);
  
  /* Container widths */
  --container: 1040px;
  --container-wide: 1200px;
  --container-narrow: 800px;
  
  /* Section spacing */
  --section-spacing: clamp(40px, 8vw, 80px);
  --section-spacing-sm: clamp(24px, 4vw, 48px);
  --section-spacing-lg: clamp(64px, 12vw, 120px);
  
  /* Formulas (computed values) */
  --container-width: min(100% - var(--gutter) * 2, var(--container));
  --container-wide-width: min(100% - var(--gutter) * 2, var(--container-wide));
}

General Recommendations

  • Use margin-block for vertical spacing between sections
  • Use margin-inline: auto for centering containers
  • For full-bleed sections use width: 100vw; margin-inline: calc(50% - 50vw);
  • Test layout on text zoom (200%) and narrow viewports (320px)
  • Avoid fixed heights; use min-height when needed

Example: Adaptive Gutters with clamp

/* Good: smoothly changes from 16px to 40px */
padding-inline: clamp(16px, 4vw, 40px);

/* Alternative: never less than 16px */
padding-inline: max(16px, 4vw);

/* Bad: sharp breakpoints */
padding-inline: 16px;
@media (min-width: 768px) {
  padding-inline: 24px;
}
@media (min-width: 1024px) {
  padding-inline: 40px;
}

BEM Naming

BEM is the default naming system. It creates predictable, low-specificity selectors and supports component reuse.

Rules

  • Block: .news-item, .service-card
  • Element: .news-item__date, .service-card__title
  • Modifier: .button--primary, .card--warning
  • Element nesting depth: max 2 levels (avoid __a__b__c)
  • Avoid modifier chains like .box--white--big unless there is a documented reason

Selectors & Specificity

Keep selectors simple. Prefer classes over tags and avoid IDs as styling hooks. Descendant selectors are allowed only when they model a true parent‚Äìchild relationship within a component.

Preferred

Low risk 
.service-card { ... }
.service-card__title { ... }
.service-card--featured { ... }

Avoid

High specificity 
#main .content .service-card .title { ... }
div > ul > li > a { ... }

Third-party overrides

Exceptions 
/* Scoped overrides for plugin markup */
.plugin-slider .swiper-button-next { ... }

Specificity strategy

  • Keep component selectors at 0-1-0 (one class) whenever possible
  • Prefer modifiers over overrides
  • Use @layer overrides for rare project-level exceptions

Formatting & Code Style

Follow a consistent formatting style. Use automated tools (Prettier + Stylelint) to enforce rules. Prefer readable CSS over overly-compressed ‚Äúclever‚Äù tricks.

Recommended formatting tools (Prettier + Stylelint)

Automated formatting reduces review noise and prevents inconsistent style across SCSS files. Keep the rules minimal and let tools do the work.

  • Run Prettier for formatting; run Stylelint for SCSS/CSS correctness and conventions.
  • Use the same versions and configs across the team (pin in package.json).
  • Do not hand-format generated CSS (assets/css/). Lint/format sources only (assets/scss/). 
npm i -D prettier stylelint stylelint-config-standard-scss stylelint-config-prettier-scss
// .prettierrc
{
  "singleQuote": true,
  "printWidth": 100
}

// .stylelintrc.cjs
module.exports = {
  extends: [
    'stylelint-config-standard-scss',
    'stylelint-config-prettier-scss'
  ],
  rules: {
    'selector-class-pattern': [
      '^[a-z][a-z0-9-]*(?:__(?:[a-z0-9-]+))?(?:--(?:[a-z0-9-]+))?$',
      { message: 'Use BEM: block__element--modifier (kebab-case)' }
    ]
  }
};

// package.json (scripts)
{
  "scripts": {
    "format": "prettier -w \"assets/scss/**/*.scss\"",
    "lint:css": "stylelint \"assets/scss/**/*.scss\""
  }
}
General rules
  • Do not overuse descendant selectors (plugin overrides are exceptions)
  • Do not add units to zero (0, not 0px)
  • Optional: omit leading zero for decimals (.5 instead of 0.5)
  • Prefer shorthand only when it does not unintentionally override existing sub-properties
  • Keep one selector per line when grouping selectors
/* Good */
.container-a,
.container-b {
  width: min(960px, 100%);
  margin: 0 auto;
  padding: 0 16px 20px;
}

/* Avoid */
.container-a, .container-b{width:960px;margin:0 auto;padding-top:0;padding-right:10px;}
Property order (recommended)

Use a consistent order to reduce review friction. 

.card {
  /* Layout / positioning */
  display: grid;
  position: relative;

  /* Box model */
  gap: 12px;
  padding: 16px;
  border-radius: 12px;

  /* Typography */
  font: 500 14px/1.4 var(--font-sans);

  /* Visual */
  background: color-mix(in srgb, var(--color-bg) 90%, white 10%);
  box-shadow: 0 8px 24px rgba(0, 0, 0, .25);

  /* Interaction */
  transition: transform .15s ease;
}

Responsive Strategy

Default to mobile-first styles with min-width breakpoints and use container queries where they provide better component-level responsiveness. Legacy breakpoint values (e.g., Bootstrap-like widths) can be used when the project requires it, but stay consistent within a project.

Breakpoints

/* Recommended: mobile-first */
@media (min-width: 576px) { /* small */ }
@media (min-width: 768px) { /* medium */ }
@media (min-width: 992px) { /* large */ }
@media (min-width: 1200px) { /* xl */ }

Legacy (max-width) style exists in older guidelines. Use only if the codebase is already written that way.

Container queries (preferred for components)

/* Component becomes responsive to its container, not the viewport */
.service-grid {
  container-type: inline-size;
}

@container (min-width: 680px) {
  .service-card {
    grid-template-columns: 160px 1fr;
  }
}

Modern units & safe viewport

  • Prefer dvh/svh/lvh for mobile viewport stability where needed
  • Prefer rem for font sizes and spacing when tied to user zoom settings
  • Use logical properties when supporting RTL or dynamic layouts (margin-inline, padding-block)

Accessibility & Performance

CSS must respect user preferences and avoid performance pitfalls. Prioritize predictable layouts and avoid heavy paint/reflow.

Motion and user preferences 
@media (prefers-reduced-motion: reduce) {
  * {
    scroll-behavior: auto !important;
    transition-duration: .01ms !important;
    animation-duration: .01ms !important;
    animation-iteration-count: 1 !important;
  }
}
Font loading 
@font-face {
  font-family: "Outfit";
  src: url("./assets/fonts/Outfit-VariableFont.woff2") format("woff2");
  font-display: swap;
}
Avoid layout thrash
  • Prefer CSS for animations; avoid JS-driven layout reads/writes per frame
  • Avoid animating width/height/top; prefer transform and opacity
  • Set image dimensions in HTML to reduce CLS

Implementation Checklist

Before you ship

  • All new components follow BEM and live in the correct folder
  • No deep nesting or ID-based styling hooks
  • Tokens are used instead of hard-coded values for repeated design decisions
  • Responsive behavior verified at key widths and with text zoom
  • Utilities are used sparingly and only for documented patterns
  • SCSS sources live under assets/scss/ and compile output is written to assets/css/ (HTML links to assets/css/style.css)
  • Live Sass Compiler workspace settings are committed (or documented) so output paths and formats are consistent for all contributors
  • If PostCSS/Autoprefixer is required, it is part of the build (not a manual editor-only step)

Rule: one block per component. Elements (__) never appear without their block. Modifiers (--) must not introduce new DOM structure, only visual or behavioral variations.

Classic Approach: .section + .section__inner

Traditional BEM-based approach with explicit wrapper classes. Each section has a .section wrapper and a .section__inner container.

CSS

/* _layout-classic.scss */
.section {
  margin-block: var(--section-spacing);
}

.section__inner {
  width: var(--container-width);
  margin-inline: auto;
  padding-inline: var(--gutter);
}

/* Modifiers */
.section--compact {
  margin-block: var(--section-spacing-sm);
}

.section--spacious {
  margin-block: var(--section-spacing-lg);
}

.section--full {
  width: 100%;
  padding-inline: var(--gutter);
}

.section--full .section__inner {
  width: min(100%, var(--container));
  padding-inline: 0;
}

.section--wide .section__inner {
  width: var(--container-wide-width);
}

.section--narrow .section__inner {
  width: min(100% - var(--gutter) * 2, var(--container-narrow));
}

HTML

<!-- Basic section -->
<section class="section">
  <div class="section__inner">
    <h2 class="section__title">Section Title</h2>
    <div class="section__content">
      <!-- Content -->
    </div>
  </div>
</section>

<!-- Compact variant -->
<section class="section section--compact">
  <div class="section__inner">
    <!-- Content -->
  </div>
</section>

<!-- Full-width with background -->
<section class="section section--full bg-gray">
  <div class="section__inner">
    <!-- Content inside container -->
  </div>
</section>

<!-- Wide container -->
<section class="section section--wide">
  <div class="section__inner">
    <!-- Wide content -->
  </div>
</section>

Pros & Cons

‚úì Pros

  • Explicit, understandable structure
  • Easy to read in DevTools
  • Flexible modifiers
  • Good legacy support

‚úó Cons

  • Extra div for each section
  • Many classes in HTML
  • Repetitive structure

CSS Grid Approach: Page-Wide Grid

Modern approach using CSS Grid for the entire page layout. Elements automatically align to grid tracks, reducing wrapper divs.

CSS

/* _layout-grid.scss */
.page-layout {
  display: grid;
  grid-template-columns:
    [full-start] minmax(var(--gutter), 1fr)
    [content-start] min(100% - var(--gutter) * 2, var(--container))
    [content-end] minmax(var(--gutter), 1fr)
    [full-end];
  grid-template-rows: auto 1fr auto;
  min-height: 100vh;
}

/* All direct children use content columns */
.page-layout > * {
  grid-column: content;
  margin-block: var(--section-spacing);
}

/* Full-width elements */
.full-bleed {
  grid-column: full;
  display: grid;
  grid-template-columns: inherit;
  margin-block: var(--section-spacing);
}

.full-bleed > * {
  grid-column: content;
}

/* Narrow container */
.content-narrow {
  grid-column: 2 / span 1;
  width: min(100%, var(--container-narrow));
  justify-self: center;
}

/* Spacing variants */
[data-spacing="compact"] {
  margin-block: var(--section-spacing-sm);
}

[data-spacing="spacious"] {
  margin-block: var(--section-spacing-lg);
}

[data-spacing="none"] {
  margin-block: 0;
}

HTML

<body class="page-layout">
  <header class="full-bleed">
    <div class="header__inner">
      <!-- Header content -->
    </div>
  </header>

  <main>
    <section>
      <h2>Regular Section</h2>
      <!-- Automatically in container -->
    </section>

    <section class="full-bleed bg-gray" data-spacing="compact">
      <div class="content-narrow">
        <!-- Narrow content on full-width background -->
      </div>
    </section>

    <section data-spacing="spacious">
      <!-- Section with large spacing -->
    </section>
  </main>

  <footer class="full-bleed">
    <div>
      <!-- Footer content -->
    </div>
  </footer>
</body>

Pros & Cons

‚úì Pros

  • Minimal HTML (no extra wrappers)
  • Centralized layout control
  • Easy full-bleed elements
  • Modern, powerful approach

‚úó Cons

  • Steeper learning curve for CSS Grid beginners
  • Less flexibility for individual elements
  • Requires mindset shift

Hybrid Approach: .container + Data Attributes

Flexible approach combining base .container class with data attributes for modifiers. Works with any HTML element.

CSS

/* _layout-hybrid.scss */
/* Base container */
.container {
  width: var(--container-width);
  margin-inline: auto;
  padding-inline: var(--gutter);
}

/* Modifiers via data attributes */
[data-container="wide"] .container {
  width: var(--container-wide-width);
}

[data-container="narrow"] .container {
  width: min(100% - var(--gutter) * 2, var(--container-narrow));
}

[data-container="full"] .container {
  width: min(100%, var(--container));
  padding-inline: 0;
}

/* Full-bleed sections */
[data-layout="full-bleed"] {
  width: 100vw;
  margin-inline: calc(50% - 50vw);
  padding-inline: var(--gutter);
}

[data-layout="full-bleed"] .container {
  width: min(100%, var(--container));
  padding-inline: 0;
}

/* Section spacing */
section, [data-layout="section"] {
  margin-block: var(--section-spacing);
}

[data-spacing="compact"] {
  margin-block: var(--section-spacing-sm);
}

[data-spacing="spacious"] {
  margin-block: var(--section-spacing-lg);
}

[data-spacing="none"] {
  margin-block: 0;
}

HTML

<!-- Basic section -->
<section>
  <div class="container">
    <h2>Section Title</h2>
    <!-- Content -->
  </div>
</section>

<!-- Wide container -->
<section data-container="wide">
  <div class="container">
    <!-- Wide content -->
  </div>
</section>

<!-- Full-bleed with background -->
<section data-layout="full-bleed" class="bg-gray">
  <div class="container" data-container="narrow">
    <!-- Narrow content on full width -->
  </div>
</section>

<!-- Compact section -->
<section data-spacing="compact">
  <div class="container">
    <!-- Content with small spacing -->
  </div>
</section>

<!-- Any element can be a container -->
<div class="container" data-container="wide">
  <!-- Content without section wrapper -->
</div>

Pros & Cons

‚úì Pros

  • Minimal HTML (.container only where needed)
  • Maximum flexibility with data attributes
  • Works with any elements
  • Easy to extend

‚úó Cons

  • Many data attributes can become confusing
  • Less explicit structure than classic approach
  • Requires team discipline
