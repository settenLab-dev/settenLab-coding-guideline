Documentation rules (Web development)

Overview

For web projects, documentation exists to keep the delivery loop short: run locally, ship safely, debug production issues quickly, and maintain UI consistency. Docs should be actionable, searchable, and updated in the same PR that changes behavior.

Core Principles

  • Docs are part of the deliverable: changes without docs are incomplete
  • Closest to the code: keep technical docs versioned with the repo
  • Single source of truth: avoid duplicated instructions and competing pages
  • Examples > prose: prefer snippets, screenshots, and checklists
  • Operational readiness: ‚Äúhow to debug‚Äù is as important as ‚Äúhow to build‚Äù

What to Document (Web)

Document the things that repeatedly slow teams down: local setup, environment variables, component usage, API contracts, build/deploy steps, and operational runbooks.

Local development

Node version, package manager, scripts, env vars, mock data, proxy settings, known pitfalls.

UI & components

Component API, usage examples, variants, accessibility notes, and design tokens.

Build / deploy / ops

  • Ephemeral Slack/Chat decisions (capture as ADRs when they matter)
  • Implementation details that are obvious in code (unless they are a known trap)

Prefer a predictable structure. Repo docs should answer 80% of questions without external links. If you use a wiki, keep an index in the repo that points to canonical pages.

Ownership

  • Every doc has an owner (team/role). If the owner changes, update the header or docs index.
  • Docs with no owner are effectively unmaintained.
  • For ‚Äúmust-not-break‚Äù docs (setup, releases), validate at least quarterly.

Style & Quality Bar

The quality bar is ‚Äúa new developer can follow it without asking questions‚Äù. If a doc requires tribal knowledge, it is not done.

Structure
  • Start with goal and prerequisites, then steps.
  • Use headings that match search intent (‚ÄúHow to run‚Äù, ‚ÄúHow to deploy‚Äù, ‚ÄúTroubleshooting‚Äù).
  • Prefer checklists and tables for repeatable procedures.
  • Link to related docs rather than duplicating content.
Examples
  • Include exact commands (npm/pnpm/yarn) and expected outputs when helpful.
  • For UI changes, include screenshots for key states (loading/empty/error).
  • For API changes, include request/response examples and error handling notes.
Freshness
  • Update docs in the same PR that changes behavior, configs, routes, or environment variables.
  • Avoid fake timestamps; instead keep docs accurate via PR review discipline.
  • For operational docs, re-run the steps periodically (quarterly minimum).

UI & Component Documentation

In web development, UI drift is one of the biggest long-term costs. Component docs should make it easy to reuse the right patterns and avoid copy/paste forks.

Component API

Props/slots, variants, defaults, and examples for common cases.

Accessibility notes

Keyboard behavior, focus management, ARIA, reduced motion, contrast considerations.

Do / Don‚Äôt

Correct usage patterns and common anti-patterns with short explanations.

Minimum for a new shared component

  • One canonical implementation location (no duplicates across features).
  • Usage examples for typical scenarios and edge cases.
  • Accessibility behavior documented (keyboard/focus/ARIA).
  • Performance considerations (virtualization for lists, memoization, etc.).

Engineering Docs (Web)

Keep ‚Äúhow the app works‚Äù concise and practical. The goal is to reduce bug cycles and simplify change.

API contracts Endpoints, schemas, auth, error categories Prevents contract drift and fragile UI code
Error handling Standard error object + UI states Consistent user experience, easier debugging
Performance budgets Bundle limits, Web Vitals targets, audit steps Prevents silent regressions
Security baseline Auth/storage rules, redaction, headers/CSP Safer defaults and better reviews

Operations, Releases & Troubleshooting

Web apps fail in production for predictable reasons: bad env vars, caching mistakes, API issues, third-party outages, and regressions. Runbooks should make diagnosis quick and repeatable.

Release documentation
  • Build command and artifact location.
  • Environment variables required per environment.
  • CDN/cache header expectations (hashed assets vs HTML).
  • Rollback steps (previous release / feature flag).
Runbook template ‚Üí
Troubleshooting requirements
  • Where to find logs/metrics (RUM, errors, API latency).
  • How to reproduce locally (flags, test accounts, mock endpoints).
  • Known failure modes (CORS, cache poisoning, auth refresh loops).
  • Escalation path (who owns hosting/CDN/API integrations).

SEO & Analytics Documentation

For many web projects, SEO and analytics are critical requirements. Document them so they remain correct through refactors and redesigns.

SEO checklist to document (as applicable)

  • Routing strategy (SPA/SSR/SSG), canonical URLs, redirects.
  • Metadata rules (title/description, OpenGraph, structured data).
  • Sitemap/robots policy and generation process.
  • Internationalization: hreflang rules if used.

Analytics rules

  • Event names and payload schema must be documented (avoid ‚Äúmystery events‚Äù).
  • Consent/Privacy rules: what is allowed to be collected.
  • Third-party tags must have an owner and route gating.

ADRs (Architecture Decision Records)

ADRs capture why major decisions were made: framework choices, routing strategy, auth model, caching approach, build tooling, or design system architecture.

When to write

Decisions that are expensive to reverse or likely to be revisited.

Where

docs/architecture/adr/, versioned with the repository.

Template

Use a consistent structure: context, decision, alternatives, consequences.

Backlog (Nulab) Workflow Rules

This section defines how developers and reviewers update tasks in Nulab (Backlog): ownership, status transitions, and mandatory comments. The backlog list itself stays in Nulab; this page defines the policy.

Policy summary

  • Creator ensures the task is understandable (context + acceptance criteria).
  • Assignee keeps status accurate and posts updates (start / blocked / handoff / done).
  • Close only after DoD: PR linked, tests green, verification notes.
  • Re-open only with a reason + reproduction steps + expected vs actual.

  • Tasks stay ‚ÄúIn Progress‚Äù with no updates for days.
  • Closing tasks without PR/test/verification notes.
  • Re-opening tasks without reproducible steps.

Documentation Checklist (Web)

When a PR changes behavior

  • README or docs index updated (if entrypoint or scripts change).
  • New environment variables documented (name, purpose, default, environment scope).
  • Component docs updated (if UI API changes).
  • Runbooks updated (if deploy/ops behavior changes).
  • SEO/analytics docs updated (if routing/metadata/events change).

Quality gates

  • No duplicated sources of truth (link instead of copy).
  • Commands verified (copy/paste works).
  • Clear ownership and escalation path for ops docs.
  • Templates provided for recurring formats (PR notes, runbooks, components).
