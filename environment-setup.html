Environment setup

Overview

This page establishes the foundation for consistent version control and development practices. The primary goal is to achieve predictable Git workflows and reliable code formatting across the entire team.

Primary Goal

Achieve consistency in version control from day one. Your team should commit, branch, and format code predictably‚Äîregardless of individual editor preferences.

  • Every developer commits with proper identity and line endings
  • Repositories follow consistent naming patterns
  • Branches and commits are structured and reviewable
  • Code formatting is automated and editor-agnostic

Setup Flow

Follow this sequence to establish a consistent foundation. Critical steps ensure team alignment; recommended steps improve quality of life.

1. Git workflow CRITICAL

Workflow, access, and local Git baseline (identity, line endings).

2. Repo baseline CRITICAL

Minimal repository baseline (README, ignore rules if applicable, lockfile if applicable).

3. Naming & branching CRITICAL

Naming conventions and branch strategy for parallel work.

4. Editor & tooling RECOMMENDED

Recommended editor/tooling defaults; flexibility is allowed.

5. Quality gates OPTIONAL

Optional automation: pre-commit hooks and/or CI where it adds value.

1. Git (Workflow Baseline)

This section describes the expected Git workflow: who initializes the repository, how contributors join, and how parallel work is organized. Tool installation details are intentionally out of scope.

Baseline: Task Owner Responsibilities

  • Create (or select) the repository and define the default branch (main).
  • Establish the initial repo baseline (README, ignore rules if applicable, lockfile if applicable).
  • Publish naming and branching expectations for the task (pages/features ownership, shared foundations).
  • Link to PR rules for review and merge requirements.

Baseline: How Contributors Join

  • Request / receive access to the repository.
  • Clone the repository and start from the latest default branch.
  • Work only in a feature branch; the default branch is kept clean.

Branch Strategy for Parallel Work

  • One task (or deliverable) = one branch. Example: one developer creates specific pages while another works on different pages.
  • Use a dedicated ‚Äúbase‚Äù branch/PR for shared foundations (layout, routing, header/footer) before page-specific branches to reduce conflicts.
  • Keep branches small and reviewable; open PRs early if coordination is required.

Local Git Settings (Expected)

  • Configure commit identity (user.name, user.email) so authorship is correct.
  • Align line endings to avoid noisy diffs (CRLF/LF). Prefer repo-level rules (.gitattributes) when present.
  • SSH authentication is recommended for convenience and security.

Common Pitfall

If line endings are not aligned, Git may show ‚Äúentire file changed‚Äù diffs even when content did not meaningfully change. Fixing this early prevents painful reviews and merge conflicts.

2. Repository Setup

Repository setup varies by project size. The goal is to provide just enough structure for clean collaboration without forcing a single template across all projects.

Always (Any Project)

  • README.md: what the project is, how to run/build/deploy, and key commands or links.
  • A clear default branch (main) and a PR-based workflow (see PR rules).

If Applicable (Based on Stack / Tooling)

  • .gitignore: include only what the project actually generates (e.g., dependency folders, build outputs, IDE files, local env files).
  • Lockfile: package-lock.json, pnpm-lock.yaml, or yarn.lock (only if a package manager is used).
  • Tooling configs (Prettier / ESLint / Stylelint / TypeScript) when the repo uses those tools.

Small static landing pages may not need a .gitignore if no generated files are produced. If you install dependencies or generate outputs, add a minimal ignore file.

Recommended

  • .gitattributes to standardize line endings and treat binaries correctly (reduces noisy diffs across OSes).
  • Documented ownership for shared code (layout/components) vs feature-specific areas to minimize conflicts.

3. Naming Conventions

Consistent naming makes repositories searchable and predictable across teams. These conventions reduce cognitive load and improve collaboration.

Repository Names

Use kebab-case with explicit scope:

  • client-product-frontend
  • brand-campaign-landing
  • company-docs-site

‚ùå Avoid vague names: new-site, project2024, website

Branch Names

Use type prefixes:

  • feat/hero-carousel
  • fix/header-zindex
  • chore/setup-eslint
  • docs/api-guide

Commit Messages

Imperative mood + conventional prefixes:

  • feat: add pricing section
  • fix: prevent layout shift
  • chore: configure prettier
  • docs: update deployment

Files & Folders

Kebab-case, descriptive nouns:

  • hero-banner.js
  • pricing-card.css
  • components/accordion/
  • utils/format-date.js

Project Structure (Flexible)

Project size and requirements vary (from small landing pages to full applications), so folder structure may differ. Where it helps consistency, we reuse familiar directories such as assets/ and components/. Introduce additional folders only when the project needs them.

Commit Message Format

Structure: type: short description (under 72 chars)

  • feat: new feature
  • fix: bug fix
  • chore: tooling, refactors, maintenance
  • docs: documentation updates
  • refactor: code improvement without feature changes 
feat: add pricing section with CTA
fix: prevent layout shift on hero image
chore: configure stylelint and prettier
docs: update deployment instructions

4. Editor Setup

Use any editor you're productive in. The key is configuring it to respect project conventions defined in .editorconfig and project config files.

Key Principle

Editor choice is personal. We provide recommended defaults (VS Code settings, formatters, and linters) to keep output consistent across the team. You may use different editors or equivalent tooling as long as the resulting code matches the repository standards (and any enabled CI checks).

üí† VS Code Setup

Popular choice with excellent extension ecosystem.

Recommended Extensions:

  • Prettier - Code formatter
  • ESLint
  • Stylelint
  • EditorConfig for VS Code
  • Path Intellisense
  • Auto Rename Tag

Workspace Settings:

Save in .vscode/settings.json (commit to repository): 

{
  "editor.formatOnSave": true,
  "editor.tabSize": 2,
  "editor.insertSpaces": true,
  "files.insertFinalNewline": true,
  "files.trimTrailingWhitespace": true,
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": true,
    "source.fixAll.stylelint": true
  }
}
üß† WebStorm / IntelliJ Setup

Powerful IDE with built-in tools and excellent refactoring support.

Configuration:

  • Enable Prettier in: Preferences ‚Üí Languages & Frameworks ‚Üí JavaScript ‚Üí Prettier
  • Set "On save" action for automatic formatting
  • Enable ESLint in: Preferences ‚Üí Languages & Frameworks ‚Üí JavaScript ‚Üí Code Quality Tools ‚Üí ESLint
  • EditorConfig is supported natively (reads .editorconfig automatically)
  • Configure file watchers for SCSS compilation if needed
‚ö° Sublime Text / Vim / Other

Lightweight and fast editors. Require manual plugin setup.

Sublime Text:

  • Install EditorConfig package via Package Control
  • Install JsPrettier for formatting
  • Install SublimeLinter with ESLint/Stylelint plugins

Vim / Neovim:

  • Install editorconfig-vim plugin
  • Use ALE or COC for linting and formatting
  • Configure to run Prettier on save

Any Editor:

  • Run Prettier via command line or watch script
  • Integrate ESLint/Stylelint with available plugins
  • Ensure line endings match .editorconfig

SCSS Workflow (Team Recommendation)

Our team uses Live Sass Compiler by Glenn Marks for VS Code, but this is optional and editor-specific. Any SCSS compiler that outputs standard CSS is acceptable.

If You Use SCSS:

  • Write styles in .scss files
  • Compile to /assets/css/ or project CSS directory
  • Commit compiled CSS to repository
  • Don't manually edit compiled CSS

Alternative Compilers:

  • Dart Sass CLI (official, recommended)
  • Build tools: Webpack, Vite, Parcel
  • WebStorm built-in file watchers
  • Any SCSS compiler that outputs standard CSS
Recommended Outputs compiled CSS to /assets/css/ to match the project file structure.

5. Quality Gates

Automate formatting and linting to enforce consistency without manual effort. Quality gates should run locally (pre-commit) and remotely (CI) so issues are caught before code review and never reach production.

Pre-commit hooks RECOMMENDED

Optional local automation to catch issues before a commit.

CI checks OPTIONAL

Run checks on PRs when CI is enabled for a repository.

Branch protection RECOMMENDED

Protect the default branch and require reviews/checks as needed.

PR template RECOMMENDED

A lightweight PR template to keep reviews consistent.

Recommended Baseline Tooling

The exact tooling depends on the project type. For static sites, you can keep it minimal; for larger codebases, standard ize on Prettier + ESLint + Stylelint.

Minimal (static projects)

Use Prettier as the only formatter. This is the fastest path to consistent diffs. 

# Install
npm i -D prettier

# Run
npx prettier . --check
npx prettier . --write

Tip: If you do not use Node tooling, enforce consistency via .editorconfig and manual formatting rules, but you will lose automation.

Team Standard (Prettier + ESLint + Stylelint)

Recommended for any project with JavaScript and CSS/SCSS. Keeps rules enforceable and predictable across editors. 

# Install
npm i -D prettier eslint stylelint lint-staged husky

# (Optional) for SCSS
npm i -D stylelint-scss

Suggested npm scripts 

{
  "scripts": {
    "format": "prettier . --write",
    "format:check": "prettier . --check",
    "lint:js": "eslint .",
    "lint:css": "stylelint \"**/*.{css,scss}\"",
    "lint": "npm run lint:js && npm run lint:css"
  }
}

Pre-commit Hooks (Husky + lint-staged)

Recommended: keep hooks fast. Run checks only on staged files (not the entire repository) to avoid slowing down commits.

Setup commands 
# One-time setup
npm i -D husky lint-staged
npx husky init

# Then edit .husky/pre-commit to run lint-staged
Example lint-staged config

Put in package.json or .lintstagedrc: 

{
  "lint-staged": {
    "*.{html,md,css,scss,js,json}": [
      "prettier --write"
    ],
    "*.{js}": [
      "eslint --fix"
    ],
    "*.{css,scss}": [
      "stylelint --fix"
    ]
  }
}
What to enforce (baseline)
  • Formatting: Prettier on all text files
  • Linting: ESLint for JS, Stylelint for CSS/SCSS
  • No commit of build outputs unless the project requires it
  • Fail fast: block commits when checks fail

CI Example (GitHub Actions)

If CI is enabled for a repository, it should run the same checks as local scripts to avoid drift. For reproducible results, use lockfiles and a pinned Node version. CI is not yet standardized across all projects‚Äîenable it when the project benefits from automated gates. 

# .github/workflows/ci.yml
name: CI
on:
  pull_request:
  push:
    branches: [ "main" ]

jobs:
  checks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - run: npm ci
      - run: npm run format:check
      - run: npm run lint

If you use pnpm or yarn, switch the install step accordingly.

Branch Protection (Recommended Settings)

  • Require PRs (no direct pushes to main)
  • Require status checks to pass before merging
  • Require at least 1 reviewer approval
  • Require conversation resolution
  • Enable ‚ÄúDismiss stale approvals‚Äù when new commits are pushed

Quick Verification Checklist

  • Git identity set: git config --global user.name
  • Line endings OK: no ‚Äúentire file changed‚Äù diffs
  • Repo has baseline files: README.md, .gitignore, .editorconfig
  • Formatter works: npm run format (or equivalent)
  • Hooks active: commit fails when rules fail
  • CI matches local: same scripts in pipeline

Recommended: .vscode/settings.json for Live Sass Compiler (Glenn Marks)

This is a project-level (workspace) configuration. It keeps SCSS source files in /assets/scss/ and writes compiled CSS to /assets/css/.

Copy & paste

Save as .vscode/settings.json in the repository root. Commit it if your team uses VS Code. 

{
  "liveServer.settings.port": 5501,
  "liveSassCompile.settings.formats": [
    {
      "format": "expanded",
      "extensionName": ".css",
      "savePath": "./assets/css",
      "generateMap": true
    },
    {
      "format": "compressed",
      "extensionName": ".min.css",
      "savePath": "./assets/css",
      "generateMap": false
    }
  ],
  "liveSassCompile.settings.excludeList": [
    "**/node_modules/**",
    "**/.vscode/**",
    "**/.history/**"
  ],
  "liveSassCompile.settings.autoprefix": [
    "> 1%",
    "last 2 versions"
  ]
}

Start compilation: Command Palette ‚Üí Live Sass: Watch Sass
Stop compilation: Command Palette ‚Üí Live Sass: Stop Watching Sass

Notes:
‚Ä¢ Do not edit /assets/css/style.css manually‚Äîtreat it as a build output.
‚Ä¢ If your project uses CI/CD, you may replace this workflow with Dart Sass CLI or a bundler.
