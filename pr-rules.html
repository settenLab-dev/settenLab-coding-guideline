PR Rules / Review Checklist (Web development)

Overview

Web PRs often fail for the same reasons: missing UI states, accessibility regressions, silent performance hits, and incomplete release notes. These rules focus reviews on user-visible correctness and operational safety.

Core Principles

  • Small and reviewable: one outcome per PR
  • Evidence-driven: screenshots, videos, test results, audits
  • Web quality gates: accessibility + performance + SEO where relevant
  • Release-safe: flags, rollback path, and monitoring notes for risky changes
  • Traceable work: link the Nulab task and decision context

PR Scope & Structure

Keep PRs focused and predictable. Large web diffs (styling + component changes + routing) are hard to review and easy to regress.

One goal

UI Evidence (Required for UI changes)

UI changes must show proof. Reviewers should be able to validate behavior without pulling the branch.

What changed Required evidence Notes
Layout / styling Before/After screenshots Include mobile + desktop breakpoints
Interactive UI Short video or GIF Show transitions and edge states
Forms Screenshots of validation + errors Loading, disabled, success, failure states
New route/page Key states + navigation path Also document SEO/metadata if applicable

Minimum UI states to validate

  • Loading (slow network)
  • Empty (no data)
  • Error (API fails)
  • Permission denied (if access control exists)
  • Responsive (mobile + desktop)

Web Quality Gates

Web projects require extra non-functional checks. If a gate is skipped, explain why in the PR.

Accessibility (a11y)
  • Keyboard navigation works (Tab order, focus visible, no focus traps unless intentional).
  • Interactive controls have accessible names (labels/aria-label).
  • Semantic HTML used where possible (buttons/links/headings).
  • Modals/drawers: focus management and aria attributes are correct.
  • Color contrast and reduced motion behavior are considered.
Performance (web-specific)
  • Bundle impact checked when adding dependencies or expanding route code.
  • Images are responsive and optimized (sizes, formats, lazy-loading where appropriate).
  • Avoid hydration/render loops; memoize expensive selectors/components when needed.
  • Web Vitals regression risk is called out (LCP/INP/CLS).

SEO (if public pages)
  • Title/description and canonical URLs are correct.
  • Structured data and OpenGraph metadata updated (if used).
  • Redirects and route changes documented to avoid breaking indexed URLs.
  • Sitemap/robots impact considered.
Analytics (if instrumented)
  • Event names/payload schema documented; avoid ‚Äúmystery events‚Äù.
  • No PII is sent; consent/opt-in logic respected.
  • Third-party tags are gated and have an owner.

Author Checklist (Web)

Before requesting review, authors must ensure the change is testable, observable, and safe to release.

Must-have
  • Nulab task linked (or explanation why not).
  • PR description includes scope, approach, and risks.
  • UI evidence included for UI changes (screenshots/video).
  • Tests updated (unit/integration/E2E as appropriate) or manual steps documented.
  • Docs updated when env vars, routes, behavior, or component APIs change.
Security & privacy
  • No secrets in code or logs; no tokens in localStorage unless explicitly approved.
  • Input validation at boundaries; no raw HTML rendering without sanitization.
  • Error messages are safe (no stack traces or internal details).
  • Analytics events do not include PII; redaction rules applied.
Cross-browser / device sanity
  • At minimum: one mobile viewport + one desktop viewport validated.
  • For critical UI: check iOS Safari behavior (scrolling, fixed elements, input focus).
  • Any known browser quirks documented for reviewers.

Reviewer Checklist (Web)

Reviewers validate user experience, correctness, and regression risk. Prefer asking questions over assuming intent; block merges when safety or testability is insufficient.

UX correctness

Are all states handled (loading/empty/error)? Is copy clear? Are interactions predictable?

Maintainability

Is it consistent with component patterns? Avoids duplication? Clear boundaries?

Non-functional

Accessibility, performance impact, privacy/analytics, SEO correctness where relevant.

Risk rubric ‚Üí

Reviewer ‚Äúmust check‚Äù items

  • Diff readability: naming, separation of concerns, no accidental churn.
  • API/error handling: consistent error model, no silent failures, retries are safe.
  • A11y basics: semantics + keyboard + focus handling for interactive components.
  • Performance: avoid unnecessary renders, large bundle additions, unbounded lists.
  • SEO/analytics: metadata/events updated and documented (if relevant).

Risk rubric (web)

  • Low: small CSS tweak, copy change, isolated component refactor with tests
  • Medium: new API call, routing changes, caching changes, new shared component
  • High: auth/session changes, analytics overhaul, SEO-impacting route changes, large dependency additions

High-risk PRs require stronger evidence: tests, feature flag, rollout plan, and monitoring notes.

CI, Approvals & Merge Rules

CI is a hard gate. If checks are flaky, treat it as a defect. Web-specific checks should not be skipped silently (lint, typecheck, unit tests, E2E for critical flows).

Minimum merge requirements

  • CI green (lint, typecheck, tests, build).
  • At least one approval (two for high-risk PRs).
  • No unresolved review threads.
  • PR description includes evidence and test notes.
  • Release/rollout notes included when user-facing.

Release safety (recommended)

  • Feature flag for risky UX or new endpoints.
  • Rollback plan (previous release + how to disable).
  • Monitoring note: what metric/log indicates success/failure.
