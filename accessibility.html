Accessibility Philosophy

Accessibility is not a feature‚Äîit's a fundamental requirement. Building accessible websites ensures everyone can use your product, regardless of ability or circumstance.

Core Principles (POUR)

  • Perceivable: Information must be presentable to users in ways they can perceive
  • Operable: Interface components must be operable by all users
  • Understandable: Information and operation must be understandable
  • Robust: Content must be robust enough to work with current and future technologies

Why Accessibility Matters

Inclusion

Human Right

15% of the world's population lives with disabilities. Accessibility ensures digital inclusion for all.

Legal

Requirement

WCAG 2.1 is required by law in many countries (ADA, AODA, EN 301 549).

Business

Smart Business

Accessible websites have better SEO, reach wider audiences, and provide better UX for everyone.

Accessibility Mindset

‚úì Proactive Approach

  • "How can we make this accessible from the start?"
  • "What barriers might users face?"
  • "Let's test with assistive technology"
  • "Accessibility is part of our definition of done"
  • "Everyone owns accessibility"

‚úó Reactive Approach

  • "We'll fix it if someone complains"
  • "It works fine for me"
  • "Screen readers are edge cases"
  • "We don't have time for accessibility"
  • "That's the QA team's responsibility"

‚ö†Ô∏è Common Misconceptions

  • "Accessibility is only for blind people" - Accessibility benefits people with visual, auditory, motor, cognitive, and situational disabilities
  • "Accessibility makes sites ugly" - Good accessibility often improves design for everyone
  • "ARIA will fix everything" - ARIA should enhance, not replace, semantic HTML
  • "Automated tools catch all issues" - Manual testing with real users is essential

Standards & Guidelines

Follow established standards to ensure compliance and provide the best experience for all users.

WCAG 2.1 Compliance Levels

Level A

Basic accessibility

  • Text alternatives for images
  • Keyboard accessible
  • No keyboard traps
  • Basic form labels
  • Minimum legal requirement

Level AA

Standard compliance

  • Color contrast 4.5:1
  • Resizable text to 200%
  • Consistent navigation
  • Error identification in forms
  • Our target standard

Level AAA

Enhanced accessibility

  • Color contrast 7:1
  • Sign language for audio
  • Extended audio description
  • No timing constraints
  • Enhanced experience

Heading Hierarchy

Proper heading structure creates a document outline for screen readers:

‚úì Correct Hierarchy 

<h1>Page Title</h1>

<h2>Introduction</h2>
<p>...</p>

<h2>Main Content</h2>
<h3>Subsection 1</h3>
<p>...</p>
<h3>Subsection 2</h3>
<p>...</p>

<h2>Conclusion</h2>
<p>...</p>

Screen readers can navigate this structure easily.

‚úó Incorrect Hierarchy 

<h1>Page Title</h1>

<h3>Introduction</h3> <!-- Skipped h2 -->
<p>...</p>

<h4>Main Content</h4> <!-- Wrong level -->
<h2>Subsection 1</h2> <!-- Jumped back -->
<p>...</p>
<h5>Subsection 2</h5> <!-- Skipped levels -->
<p>...</p>

<h6>Conclusion</h6> <!-- Too deep -->
<p>...</p>

Confusing for screen readers - creates broken document outline.

Keyboard Navigation

Many users navigate websites using only a keyboard. Ensure all interactive elements are keyboard accessible.

Keyboard Navigation Requirements

Focus Order

  • Logical tab order (DOM order)
  • No keyboard traps
  • Skip links for navigation
  • Manage focus in modals

Focus Indicators

  • Visible focus styles
  • Never remove outline:focus
  • High contrast focus rings
  • Custom focus styles allowed

Keyboard Support

  • All interactive elements focusable
  • Space/Enter for activation
  • Arrow keys for menus, sliders
  • Esc to close modals, menus

Focus Management Patterns

Skip Links

Allow keyboard users to skip repetitive navigation:

‚úì Good Skip Link 

<body>
  <a href="#main-content" 
     class="skip-link">
    Skip to main content
  </a>
  
  <nav>...</nav>
  
  <main id="main-content" tabindex="-1">
    <!-- Main content here -->
  </main>
</body>

/* CSS */
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  background: #000;
  color: white;
  padding: 8px;
  z-index: 100;
}

.skip-link:focus {
  top: 0;
}

‚úó Missing Skip Link 

<body>
  <nav>
    <!-- 20 navigation items -->
    <ul>
      <li><a href="#">Home</a></li>
      <li><a href="#">About</a></li>
      <li><a href="#">Services</a></li>
      <li><a href="#">Products</a></li>
      <li><a href="#">Contact</a></li>
      <!-- 15 more items... -->
    </ul>
  </nav>
  
  <main>
    <!-- Keyboard users must tab through 
         all navigation to reach content -->
  </main>
</body>

‚ö†Ô∏è Common Keyboard Navigation Issues

  • Hidden interactive elements that appear on hover but aren't keyboard accessible
  • Custom elements without proper tabindex (div instead of button)
  • Focus traps in modals - can't escape with keyboard
  • Removing focus outlines with outline: none without providing alternative
  • Non-logical tab order from CSS positioning (absolute/fixed)

ARIA & Screen Readers

Accessible Rich Internet Applications (ARIA) provides additional semantics for custom widgets and dynamic content.

ARIA Principles

First Rule of ARIA

Essential

Don't use ARIA if you can use native HTML. Prefer semantic elements over ARIA roles.

Second Rule

Essential

Don't change native semantics. Never add ARIA to semantic elements unless absolutely necessary.

Third Rule

Important

Make interactive elements keyboard accessible. ARIA doesn't add keyboard behavior.

Common ARIA Patterns

Pattern ARIA Usage Example
Live Regions aria-live Notifications, chat messages
Modal Dialogs role="dialog", aria-modal="true" Popup windows, alerts
Progress Bars role="progressbar", aria-valuenow File uploads, loading indicators
Tabs role="tablist", role="tab", role="tabpanel" Tabbed interfaces
Tooltips role="tooltip" Information popups
Alerts role="alert" Error messages, success notifications
Breadcrumbs aria-label="Breadcrumb" Navigation trails

ARIA Labels & Descriptions

‚úì Proper ARIA Usage 

<!-- Button with icon only -->
<button 
  aria-label="Close dialog"
  class="icon-button">
  <svg>...</svg>
</button>

<!-- Form field with error -->
<div class="form-group">
  <label for="email">Email</label>
  <input 
    id="email"
    type="email"
    aria-invalid="true"
    aria-describedby="email-error">
  <div id="email-error" role="alert">
    Please enter a valid email
  </div>
</div>

<!-- Live region for updates -->
<div 
  aria-live="polite"
  aria-atomic="true">
  <!-- Dynamic content -->
</div>

‚úó ARIA Misuse 

<!-- Redundant ARIA -->
<nav role="navigation"> <!-- Redundant -->
  <ul>...</ul>
</nav>

<!-- Overriding semantics -->
<h1 role="button">Click me</h1> <!-- Wrong! -->

<!-- Missing accessible name -->
<button class="icon-button">
  <svg>...</svg> <!-- No label -->
</button>

<!-- Inaccessible dynamic content -->
<div id="notification">
  New message received <!-- Screen reader won't announce -->
</div>

Visual Accessibility

Visual accessibility ensures content is perceivable by users with visual impairments, color blindness, or low vision.

Color Contrast Requirements

Normal Text

WCAG AA

  • 4.5:1 contrast ratio
  • For text ‚â§ 18pt
  • For text ‚â§ 14pt bold
  • Required for compliance

Large Text

WCAG AA

  • 3:1 contrast ratio
  • For text ‚â• 18pt
  • For text ‚â• 14pt bold
  • Headers, large print

Enhanced

WCAG AAA

  • 7:1 normal text
  • 4.5:1 large text
  • Better for low vision
  • Recommended where possible

Color & Meaning

Never rely on color alone to convey information:

‚úì Accessible Design 

<!-- Form validation -->
<div class="form-group">
  <label for="email">Email</label>
  <input 
    id="email"
    type="email"
    aria-invalid="true"
    aria-describedby="email-error">
  <div id="email-error" role="alert">
    <span class="error-icon" aria-hidden="true">‚ö†Ô∏è</span>
    <span>Please enter a valid email</span>
  </div>
</div>

/* CSS */
.error-icon {
  color: #dc2626; /* Red color */
}

[aria-invalid="true"] {
  border: 2px solid #dc2626;
  /* Plus icon, text, and border */
}

Uses color, icons, text, and borders to convey error state.

‚úó Color-Only Design 

<!-- Form validation -->
<div class="form-group">
  <label for="email">Email</label>
  <input id="email" type="email">
  <div class="error">
    Please enter a valid email
  </div>
</div>

/* CSS */
.error {
  color: #dc2626; /* Only color */
}

/* For color-blind users:
   - Can't distinguish error state
   - No other indicators
   - Might miss validation errors */

Relies solely on color - fails for color-blind users.

Responsive Typography

Font Size & Scaling 
/* Use relative units for font sizes */
:root {
  /* Base font size in pixels for calculation */
  font-size: 16px;
}

/* Good: Relative units */
body {
  font-size: 1rem; /* 16px */
}

h1 {
  font-size: 2.5rem; /* 40px, scales with root */
}

.small-text {
  font-size: 0.875rem; /* 14px */
}

/* Good: Responsive typography */
@media (max-width: 768px) {
  :root {
    font-size: 14px; /* Scale everything down */
  }
}

/* Bad: Fixed pixels */
.bad-example {
  font-size: 14px; /* Doesn't scale */
  line-height: 20px; /* Fixed line height */
}

/* Good: Responsive line height */
.good-example {
  font-size: 1rem;
  line-height: 1.5; /* Unitless, scales with font */
}
Text Spacing & Layout 
/* WCAG Text Spacing Success Criterion */
/* Support 200% zoom and custom spacing */

/* Good: Flexible containers */
.content {
  max-width: 65ch; /* Readable line length */
  /* Allow text to reflow */
  overflow-wrap: break-word;
  word-break: break-word;
}

/* Support text spacing adjustments */
* {
  /* These properties should not interfere with:
     - Line height to at least 1.5 times font size
     - Spacing following paragraphs to at least 2 times font size
     - Letter spacing to at least 0.12 times font size
     - Word spacing to at least 0.16 times font size
  */
}

/* Bad: Fixed heights that clip text */
.bad-container {
  height: 100px; /* Text gets clipped when zoomed */
  overflow: hidden;
}

/* Good: Minimum touch targets */
button, 
a,
input[type="checkbox"],
input[type="radio"] {
  min-height: 44px;
  min-width: 44px;
}

Interactive Components

Custom interactive components must provide equivalent functionality to native elements for all users.

Accessible Component Patterns

Custom vs Native Elements

Component Native Element Custom Implementation Requirements
Button <button> role="button", keyboard events, focus management
Checkbox <input type="checkbox"> role="checkbox", aria-checked, Space key
Radio Button <input type="radio"> role="radio", arrow key navigation, aria-checked
Select/Dropdown <select> role="listbox", arrow keys, Esc, Enter, type-ahead
Modal Dialog <dialog> role="dialog", focus trap, Esc key, backdrop click
Tab Interface None role="tablist", role="tab", arrow keys, Home/End
Tooltip title attribute role="tooltip", mouse+hover, keyboard focus, timer

‚ö†Ô∏è Common Interactive Component Issues

  • Clickable <div> instead of <button> - Missing keyboard support and semantics
  • Custom selects without proper ARIA - Screen readers can't understand the options
  • Modals that don't trap focus - Keyboard users can tab behind the modal
  • Carousels without pause controls - Difficult for users who need more time
  • Hover-only tooltips - Keyboard users can't access the information

Accessible Forms

Forms are critical for user interaction and must be accessible to everyone, including screen reader users and keyboard-only users.

Form Accessibility Requirements

Labels

Essential

Every form control must have an associated label. Use <label> or aria-label/aria-labelledby.

Instructions

Essential

Provide clear instructions and error messages. Use aria-describedby for additional information.

Validation

Important

Validate inputs and provide helpful error messages. Use aria-invalid and role="alert".

Grouping

Important

Group related controls with <fieldset> and <legend>.

‚úó Inaccessible Forms

Frameworks & Libraries

Modern frameworks and libraries can help or hinder accessibility. Choose tools that support accessible development.

Framework-Specific Accessibility

React Accessibility 
// React Accessibility Patterns
import React, { useRef, useEffect } from 'react';

// Focus management
function Modal({ isOpen, onClose }) {
  const modalRef = useRef();
  
  useEffect(() => {
    if (isOpen) {
      // Focus trap
      modalRef.current.focus();
    }
  }, [isOpen]);
  
  // Handle keyboard events
  const handleKeyDown = (e) => {
    if (e.key === 'Escape') onClose();
    if (e.key === 'Tab') {
      // Implement focus trap
    }
  };
  
  return isOpen ? (
    <div
      role="dialog"
      aria-modal="true"
      aria-labelledby="modal-title"
      onKeyDown={handleKeyDown}
      tabIndex={-1}
      ref={modalRef}
    >
      <h2 id="modal-title">Modal Title</h2>
      {/* Modal content */}
    </div>
  ) : null;
}

// Accessible forms
function LoginForm() {
  const [error, setError] = useState('');
  
  return (
    <form onSubmit={handleSubmit}>
      <label htmlFor="email">Email</label>
      <input
        id="email"
        type="email"
        aria-describedby={error ? "email-error" : undefined}
        aria-invalid={!!error}
      />
      {error && (
        <div id="email-error" role="alert">
          {error}
        </div>
      )}
    </form>
  );
}

‚ö†Ô∏è Framework Accessibility Pitfalls

  • Over-reliance on CSS-in-JS - Can break user style sheets and preferences

WCAG 2.1 Success Criteria Quick Reference

Perceivable (Guideline 1)

Code Description Level
1.1.1 Non-text Content Provide text alternatives for non-text content A
1.2.1 Audio-only and Video-only Provide alternatives for time-based media A
1.3.1 Info and Relationships Programmatically determine information and relationships A
1.3.2 Meaningful Sequence Present content in a meaningful order A
1.3.3 Sensory Characteristics Don't rely solely on sensory characteristics A
1.4.1 Use of Color Don't use color as the only visual means of conveying information A
1.4.3 Contrast (Minimum) Text and images of text have contrast ratio of at least 4.5:1 AA
1.4.4 Resize Text Text can be resized up to 200% without loss of content or function AA
1.4.10 Reflow Content can be presented without loss of information or functionality AA
1.4.11 Non-text Contrast Visual presentation of UI components and graphical objects have contrast ratio of at least 3:1 AA
1.4.12 Text Spacing No loss of content or functionality when text spacing is adjusted AA

Operable (Guideline 2)

Code Description Level
2.1.1 Keyboard All functionality operable through a keyboard interface A
2.1.2 No Keyboard Trap Keyboard focus not trapped in any component A
2.1.4 Character Key Shortcuts If character key shortcuts exist, they can be turned off or remapped A
2.2.1 Timing Adjustable Users can turn off, adjust, or extend time limits A
2.3.1 Three Flashes or Below Web pages do not contain anything that flashes more than three times per second A
2.4.1 Bypass Blocks Provide a way to bypass blocks of content repeated on multiple pages A

Modal Focus Management

// React Modal with Focus Management
import React, { useEffect, useRef } from 'react';

function Modal({ isOpen, onClose, title, children }) {
  const modalRef = useRef();
  const closeButtonRef = useRef();
  const previousFocus = useRef();
  
  useEffect(() => {
    if (isOpen) {
      // Save currently focused element
      previousFocus.current = document.activeElement;
      
      // Focus the modal
      modalRef.current.focus();
      
      // Trap focus inside modal
      const handleKeyDown = (e) => {
        if (e.key === 'Escape') {
          onClose();
        }
        
        if (e.key === 'Tab') {
          const focusableElements = modalRef.current.querySelectorAll(
            'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
          );
          
          const firstElement = focusableElements[0];
          const lastElement = focusableElements[focusableElements.length - 1];
          
          if (e.shiftKey) {
            // Shift + Tab
            if (document.activeElement === firstElement) {
              lastElement.focus();
              e.preventDefault();
            }
          } else {
            // Tab
            if (document.activeElement === lastElement) {
              firstElement.focus();
              e.preventDefault();
            }
          }
        }
      };
      
      document.addEventListener('keydown', handleKeyDown);
      
      // Return focus on close
      return () => {
        document.removeEventListener('keydown', handleKeyDown);
        if (previousFocus.current) {
          previousFocus.current.focus();
        }
      };
    }
  }, [isOpen, onClose]);
  
  if (!isOpen) return null;
  
  return (
    <div
      role="dialog"
      aria-modal="true"
      aria-labelledby="modal-title"
      tabIndex={-1}
      ref={modalRef}
      className="modal"
    >
      <div className="modal-content">
        <div className="modal-header">
          <h2 id="modal-title">{title}</h2>
          <button
            ref={closeButtonRef}
            onClick={onClose}
            aria-label="Close modal"
            className="close-button"
          >
            <span aria-hidden="true">×</span>
          </button>
        </div>
        <div className="modal-body">
          {children}
        </div>
        <div className="modal-footer">
          <button onClick={onClose}>Cancel</button>
          <button onClick={handleConfirm}>Confirm</button>
        </div>
      </div>
    </div>
  );
}

Dynamic Content Focus Management

// Focus management for dynamically added content
function Notification({ message, type = 'info', onDismiss }) {
  const notificationRef = useRef();
  
  useEffect(() => {
    // Focus the notification when it appears
    notificationRef.current.focus();
    
    // Announce to screen readers
    const announceToScreenReader = () => {
      // Using aria-live region
      const liveRegion = document.getElementById('a11y-live-region');
      if (liveRegion) {
        liveRegion.textContent = `${type} notification: ${message}`;
        setTimeout(() => {
          liveRegion.textContent = '';
        }, 1000);
      }
    };
    
    announceToScreenReader();
    
    // Auto-dismiss after 5 seconds
    const timer = setTimeout(() => {
      onDismiss();
    }, 5000);
    
    return () => clearTimeout(timer);
  }, [message, type, onDismiss]);
  
  return (
    <div
      ref={notificationRef}
      role="status"
      aria-live="polite"
      aria-atomic="true"
      tabIndex={-1}
      className={`notification notification--${type}`}
    >
      <div className="notification-content">
        <span className="notification-icon" aria-hidden="true">
          {type === 'success' ? '‚úì' : type === 'error' ? '‚úó' : '‚Ñπ'}
        </span>
        <span className="notification-message">{message}</span>
      </div>
      <button
        onClick={onDismiss}
        aria-label="Dismiss notification"
        className="notification-dismiss"
      >
        <span aria-hidden="true">×</span>
      </button>
    </div>
  );
}

Accessible Tab Interface

// Accessible Tabs Component
import React, { useState, useEffect, useRef } from 'react';

function Tabs({ tabs }) {
  const [activeTab, setActiveTab] = useState(0);
  const tablistRef = useRef();
  
  const handleKeyDown = (e) => {
    const tabCount = tabs.length;
    
    switch (e.key) {
      case 'ArrowRight':
      case 'ArrowDown':
        e.preventDefault();
        setActiveTab((prev) => (prev + 1) % tabCount);
        break;
        
      case 'ArrowLeft':
      case 'ArrowUp':
        e.preventDefault();
        setActiveTab((prev) => (prev - 1 + tabCount) % tabCount);
        break;
        
      case 'Home':
        e.preventDefault();
        setActiveTab(0);
        break;
        
      case 'End':
        e.preventDefault();
        setActiveTab(tabCount - 1);
        break;
        
      default:
        break;
    }
  };
  
  useEffect(() => {
    // Focus the active tab when it changes
    const activeTabElement = tablistRef.current?.querySelector(
      `[role="tab"][aria-selected="true"]`
    );
    if (activeTabElement) {
      activeTabElement.focus();
    }
  }, [activeTab]);
  
  return (
    <div className="tabs">
      <div
        ref={tablistRef}
        role="tablist"
        aria-label="Product information tabs"
        onKeyDown={handleKeyDown}
      >
        {tabs.map((tab, index) => (
          <button
            key={tab.id}
            id={`tab-${tab.id}`}
            role="tab"
            aria-selected={activeTab === index}
            aria-controls={`panel-${tab.id}`}
            tabIndex={activeTab === index ? 0 : -1}
            onClick={() => setActiveTab(index)}
            className={`tab ${activeTab === index ? 'active' : ''}`}
          >
            {tab.label}
            {tab.notification && (
              <span className="sr-only">
                , has notification
              </span>
            )}
          </button>
        ))}
      </div>
      
      {tabs.map((tab, index) => (
        <div
          key={tab.id}
          id={`panel-${tab.id}`}
          role="tabpanel"
          aria-labelledby={`tab-${tab.id}`}
          hidden={activeTab !== index}
          tabIndex={0}
          className="tabpanel"
        >
          {activeTab === index && tab.content}
        </div>
      ))}
    </div>
  );
}

  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}
