File Structure

Overview

A consistent file structure is required for all projects. It simplifies onboarding, maintenance, and deployment. This document defines how to organize HTML pages, CSS/JS assets, images, and components for multi-page sites.

Key principles

  • One shared assets/ folder per project.
  • Page-specific files live next to the page or in page folders.
  • Global styles/scripts are reused across all pages.
  • File names are lowercase, use kebab-case, and English identifiers.

Basic project structure (static site)

Default structure for a static multi-page project without a build system (plain HTML/CSS/JS). 

project-root/
  index.html
  about.html
  guidelines.html
  file-structure.html
  vscode.html

  assets/
    css/
      base.css
      layout.css
      components.css
      pages/
        home.css
        about.css
    js/
      main.js
      slider.js
      forms.js
      vendor/
        swiper-bundle.min.js
    images/
      global/
      pages/
        home/
        about/
        guidelines/
      icons/
      logos/
    fonts/
  • assets/css — global styles, plus pages/ for page-specific overrides.
  • assets/js — main.js and component-specific scripts.
  • assets/images — shared images, plus per-page folders.
Full structure example

Assets organization

CSS and JS assets must be grouped by responsibility: base, layout, components, pages. Vendor libraries are always isolated.

CSS structure

assets/css/
  base.css          /* resets, typography, colors, root vars */
  layout.css        /* grid, containers, layout blocks */
  components.css    /* buttons, cards, modals, sliders */
  utilities.css     /* helper classes (u-*) */
  pages/
    home.css
    about.css
    guidelines.css
  • base.css — reset + global design tokens.
  • layout.css — page shell, header, footer.
  • components.css — BEM components.
  • pages/*.css — page-specific tweaks (minimize usage).

JavaScript structure

assets/js/
  main.js          /* global init: menus, header, shared UX */
  slider.js        /* sliders / swiper initialization */
  forms.js         /* validation, form handlers */
  vendor/
    swiper-bundle.min.js
  • main.js must only initialize components and bind global events.
  • Component-specific logic (sliders, modals) lives in separate files.
  • Third-party libraries are stored in vendor/ and never modified directly.

Components & partials

If the project uses templates or includes (e.g., via server-side includes, build tools, or static site generator), keep shared blocks in a dedicated folder. 

project-root/
  components/
    header.html
    footer.html
    nav-main.html
    slider-hero.html
    product-card.html
    news-item.html
  • Each file contains one logical component.
  • Component filenames match BEM blocks: product-card.html → .product-card.
  • When using a templating engine, this folder may be under src/components/.

Recommended pattern

src/
  components/
    header/
      header.html
      header.css
      header.js
    slider/
      slider.html
      slider.css
      slider.js

For projects with a build step, group component assets by component inside src/components/, and compile into a flat assets/ folder for production.

Images & media folders

Images follow the same pattern across all projects to keep references predictable and reusable. 

assets/images/
  global/
    bg-texture.webp
    placeholder.webp
  logos/
    company-logo.svg
    brand-mark.svg
  icons/
    icon-menu.svg
    icon-close.svg
    icon-arrow-right.svg
  ui/
    btn-gradient.webp
    card-bg.webp
  pages/
    home/
      hero-main.webp
      hero-mobile.webp
      feature-01.webp
    about/
      team-01.webp
      office-01.webp
    guidelines/
      slider-example.webp
      gallery-example.webp
  • global/ — shared backgrounds, placeholders, generic visuals.
  • logos/ — brand assets, always SVG if possible.
  • icons/ — UI icons, mostly SVG.
  • pages/<page>/ — page-specific images, named according to context (hero, section, feature).

Image naming rules

  • Use English, lowercase, kebab-case.
  • Include type or usage: hero-main.webp, section-team.webp, card-feature-01.webp.
  • For responsive variants, add suffix: hero-main@2x.webp, hero-main-mobile.webp.

Folder naming rules

Folder names must be short, descriptive, and stable across projects.

Rules

  • Use English for all folder names.
  • Use kebab-case (product-list).
  • No spaces, no uppercase, no special characters.
  • Use plural for collections: assets, images, components.
  • Avoid deep nesting (> 4 levels) unless necessary (e.g. component-based src/components/...).

Examples: good vs bad

Good:
  assets/images/pages/home/
  assets/js/vendor/
  components/product-card/

Bad:
  img/
  JSFiles/
  components/newDesign2/
  画像/
