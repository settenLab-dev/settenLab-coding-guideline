File structure

Overview

A consistent file structure is essential for maintainability, onboarding, and collaboration. This guide covers organization patterns for projects of different scales‚Äîfrom simple static sites to complex applications with build systems.

Core Principles

  • Structure should scale with project complexity
  • Related files are grouped together (proximity principle)
  • Global assets are shared; page-specific assets are isolated
  • File names use kebab-case, English, lowercase
  • Naming is predictable and searchable

Choosing the Right Structure

Different projects require different organization strategies. Choose based on project size, tooling, and team needs.

Small Projects

< 10 pages, static HTML/CSS/JS

10-50 pages, may use build tools

Decision Tree

  • Plain HTML/CSS/JS? ‚Üí Use flat structure (see Small Projects)
  • Using build tools (Webpack/Vite)? ‚Üí Use src/dist separation (see Medium/Large)
  • Component library or design system? ‚Üí Use atomic/modular structure
  • Framework (React/Vue/Svelte)? ‚Üí Follow framework conventions + our naming rules

Small Projects Overview

For simple multi-page sites without build systems. HTML files at root, shared assets in assets/.

When to Use Small Structure

  • Landing pages, small marketing sites
  • Documentation sites without complex builds
  • Prototypes and quick demos
  • Projects with < 10 pages
View full structure ‚Üí

‚ö†Ô∏è Limitations

This structure doesn't scale well beyond 10-15 pages. If your project grows, consider migrating to a medium structure with build tools.

Medium Projects Overview

For projects using build tools (Webpack, Vite, Parcel) or static site generators. Separates source (src/) from output (dist/).

Key Differences from Small Projects

  • src/ contains all source files
  • dist/ is the compiled output (not committed to Git)
  • Components are grouped with their styles and scripts
  • Build tool handles concatenation, minification, optimization
View full structure ‚Üí

When to Use Medium Structure

  • 10-50 pages with shared components
  • Need for SCSS, PostCSS, or CSS preprocessing
  • Want to optimize and bundle assets automatically
  • Using modern JavaScript (ES modules, transpilation)

Large Projects Overview

For complex applications, design systems, or sites with 50+ pages. Uses feature-based or atomic organization.

When to Use Large Structure

  • Large applications (50+ pages)
  • Design systems or component libraries
  • Multiple developers working simultaneously
  • Need for clear separation of concerns
  • Reusable components across multiple projects
View full structure ‚Üí

Organization Approaches

  • Feature-based: Group by features (home/, product/, blog/)
  • Atomic Design: Group by component complexity (atoms/, molecules/, organisms/)
  • Hybrid: Combine approaches based on project needs

Assets Organization

CSS, JavaScript, images, and fonts follow consistent patterns across all project sizes.

CSS Structure

Small Projects: 

assets/css/
  style.css           /* everything in one file */
OR
  base.css            /* resets, typography, tokens */
  components.css      /* all components */
  utilities.css       /* helper classes */

Medium/Large Projects: 

src/styles/
  tokens.css          /* CSS custom properties */
  reset.css           /* normalization */
  base.css            /* typography, global styles */
  layout.css          /* grid, containers */
  utilities.css       /* u-* helper classes */
  
components/           /* component-specific CSS */
  button/button.css
  header/header.css

Page-Specific CSS:

Minimize page-specific CSS. Use only for:

  • One-off layout adjustments unique to a page
  • Temporary overrides (document why)
  • A/B test variations
JavaScript Structure

Small Projects: 

assets/js/
  main.js             /* initialization, global logic */
  slider.js           /* feature-specific */
  forms.js
  vendor/
    swiper.min.js     /* third-party, unmodified */

Medium/Large Projects: 

src/scripts/
  main.js             /* app entry point */
  components/
    slider.js
    modal.js
  utils/
    dom.js            /* DOM helpers */
    api.js            /* API calls */
    format.js         /* data formatting */
  config/
    constants.js
  vendor/
    (rarely used - prefer npm packages)

Key Rules:

  • main.js only initializes, never contains feature logic
  • Feature code lives in separate modules
  • Vendor libraries: use npm packages > local copies
  • Never modify vendor files directly

Image Naming Rules:

  • English, lowercase, kebab-case
  • Include context: hero-main.webp, section-team.webp
  • Retina: image@2x.webp, image@3x.webp
  • Responsive: image-mobile.webp, image-tablet.webp

Fonts 
assets/fonts/
  Outfit-VariableFont.woff2
  Outfit-Regular.woff2
  Outfit-Bold.woff2

Best Practices:

  • Prefer WOFF2 (best compression + broad support)
  • Use variable fonts when possible (Outfit-VariableFont.woff2)
  • Self-host critical fonts for performance
  • Preload primary fonts in <head>
  • Avoid mixing too many font families (max 2-3)

Components & Partials

When using templates, includes, or component-based architecture, organize reusable UI elements systematically.

Component Structure (Recommended)

components/
  button/
    button.html       /* or button.jsx, button.vue */
    button.css
    button.js
    button.test.js    /* if testing */
    README.md         /* component documentation */
  
  header/
    header.html
    header.css
    header.js
    header.test.js
  
  modal/
    modal.html
    modal.css
    modal.js
    modal.test.js

Each component is self-contained: all related files grouped together.

Component Naming Convention

  • Folder name matches BEM block: product-card/ ‚Üí .product-card
  • Files inside use same base name: product-card.html, product-card.css
  • Use kebab-case for multi-word components
  • Be specific: nav-main > nav, button-primary > button1

Each complex component should include a README with:

  • Purpose and usage examples
  • Props/parameters (if applicable)
  • Variants and modifiers
  • Dependencies
  • Known limitations

Configuration Files

Configuration files live at project root. They define tooling behavior and project standards.

File Purpose Required?
package.json Dependencies, scripts, project metadata If using Node.js
.gitignore Files excluded from version control Always
.editorconfig Editor-agnostic formatting rules Recommended
.prettierrc Prettier formatting rules If using Prettier
.eslintrc.js JavaScript linting rules If using ESLint
.stylelintrc.js CSS linting rules If using Stylelint
.env.example Environment variable template If using .env
README.md Project documentation Always
CHANGELOG.md Version history and changes Recommended

Environment Variables (.env)

Never commit .env to Git. Always provide .env.example: 

# .env.example
API_BASE_URL=https://api.example.com
GOOGLE_ANALYTICS_ID=UA-XXXXXXXXX-X
ENVIRONMENT=development

Developers copy .env.example to .env and fill in real values locally.

Naming Conventions

Consistent naming makes projects searchable and predictable. These rules apply to all files and folders.

Files

  • Use kebab-case
  • Lowercase only
  • English language
  • Descriptive, not abbreviated
  • hero-banner.js ‚úì
  • HeroBanner.js ‚úó

Folders

  • Use kebab-case
  • Plural for collections
  • No nesting > 4 levels
  • components/ ‚úì
  • Components/ ‚úó
  • component/ ‚úó

Assets

  • Include context in name
  • Use hyphens as separators
  • hero-main.webp ‚úì
  • hero-main@2x.webp ‚úì
  • img1.jpg ‚úó
  • hero_main.webp ‚úó

Examples: Good vs Bad

‚úì Good 

components/product-card/
assets/images/pages/home/
hero-banner.js
pricing-section.css
api-client.js

‚úó Bad 

Components/ProductCard/
img/
HeroBanner.js
pricing_section.css
apiClient.js

‚ö†Ô∏è Common Mistakes

  • Mixed cases: myComponent.js + MyOtherComponent.js
  • Abbreviations: btn.js, hdr.css (use full words)
  • Generic names: file1.js, temp.css, new-component.js
  • Non-English: Using local language characters in filenames
  • Spaces: my file.js (breaks on command line)

Tests & Documentation

Tests and documentation are first-class citizens. They need organized homes in your project structure.

Test Organization

Option 1: Co-located Tests

Tests live next to the code they test (recommended for component libraries): 

components/
  button/
    button.js
    button.css
    button.test.js    /* unit tests */
  modal/
    modal.js
    modal.css
    modal.test.js

Option 2: Separate Tests Folder

Tests mirror the src/ structure (common for applications): 

src/
  components/
    button.js
    modal.js
  utils/
    format.js

tests/              /* or __tests__/ */
  components/
    button.test.js
    modal.test.js
  utils/
    format.test.js
  e2e/
    homepage.spec.js

Test File Naming:

  • *.test.js - Unit tests (Jest convention)
  • *.spec.js - Unit tests (Mocha/Jasmine convention)
  • *.e2e.js - End-to-end tests
  • *.integration.js - Integration tests

Migrating Existing Projects

Restructuring a project takes planning. Don't try to do everything at once‚Äîprioritize high-impact changes.

Phase 1: Foundation

Add missing config files, establish naming conventions for new files

Phase 2: Assets

Reorganize CSS/JS/images into proper folders, update references

Phase 3: Components

Extract reusable components, group related files together

Migration Checklist

  • ‚úì Add .gitignore, .editorconfig, README.md
  • ‚úì Create assets/ folder structure
  • ‚úì Establish naming convention (document it in README)
  • ‚úì Move images to assets/images/
  • ‚úì Consolidate CSS files by purpose (base, components, utilities)
  • ‚úì Organize JS into logical modules
  • ‚úì Extract shared components
  • ‚úì Update all file references in HTML/CSS/JS
  • ‚úì Test thoroughly after each phase

‚ö†Ô∏è Migration Tips

  • Work incrementally: Restructure one section at a time
  • Use Git branches: Create refactor/file-structure branch
  • Keep backups: Tag current state before major changes
  • Update references: Use search/replace for path changes
  • Test continuously: Don't restructure everything then test
  • Document changes: Update README with new structure

Troubleshooting

Common issues when organizing project files and how to solve them.

Structure is too deep (> 5 levels)

Problem:

src/features/product/components/cards/product/variants/large/...

Solution:

  • Flatten the structure by removing intermediate folders
  • Use more descriptive names instead of nesting: product-card-large/
  • Question if deep nesting actually adds clarity

src/components/product-card-large/
Unclear where to put page-specific code

Guidelines:

  • If used only on one page: Put in pages/[page-name]/
  • If used on 2-3 pages: Still page-specific, keep in pages/
  • If used on 3+ pages: It's shared, move to components/
  • When in doubt: Start in pages/, refactor when reused
Different developers use different structures

Solution:

  • Document the agreed structure in README.md
  • Use this guideline as the source of truth
  • Add structure diagrams/examples to project docs
  • Code review for structure compliance
  • Use automated linting for file naming (eslint-plugin-filename-case)
Build output mixed with source files

Problem: Compiled files (.min.js, .min.css) mixed with source

Solution:

  • Always separate src/ from dist/ when using build tools
  • Add dist/ to .gitignore
  • Only commit source files to version control
  • Build output is generated, not authored
# .gitignore
dist/
build/
*.min.css
*.min.js
*.map
Too many files at root level

Problem: 20+ files at project root

What stays at root:

  • Config files (.gitignore, package.json, etc.)
  • Documentation (README.md, CHANGELOG.md)
  • Main entry HTML files (for small projects)

What moves into folders:

  • All assets ‚Üí assets/
  • Page HTML files (medium+) ‚Üí src/pages/
  • Components ‚Üí components/ or src/components/
  • Scripts ‚Üí scripts/ (build scripts, not app code)
Vendor libraries causing issues

Best Practices:

  • Prefer npm packages over local copies
  • If you must use local copies, keep in assets/js/vendor/
  • Never modify vendor files - creates update nightmares
  • Document vendor file versions in README or package.json
  • Update regularly, test after updates

npm install swiper


<script src="https://cdn.jsdelivr.net/npm/swiper@11/...">


assets/js/vendor/swiper-bundle.min.js  /* v11.0.0 */

Implementation Checklist

New Project Setup

  • Choose structure based on project size (small/medium/large)
  • Create folder structure before writing code
  • Add all config files (.gitignore, .editorconfig, etc.)
  • Write initial README with structure explanation
  • Document naming conventions
  • Set up assets folders (css/, js/, images/, fonts/)

Ongoing Maintenance

  • Review structure during code reviews
  • Refactor when files used across 3+ pages
  • Keep vendor libraries updated and documented
  • Remove unused files regularly
  • Update README when structure changes
  • Ensure new team members understand structure

Quality Gates

  • All files follow kebab-case naming
  • No files > 4 levels deep (except node_modules)
  • No orphaned files (unused assets)
  • Build output not committed to Git
  • Each component is self-contained
  • README documents current structure
