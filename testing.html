Testing

Testing Guidelines

This page describes our standard testing workflow: how to use browser DevTools (Inspect mode), test layouts on different devices, and verify cross-browser behavior before release.

Goals

  • Detect layout and responsive issues early (in DevTools).
  • Test on real and emulated devices before release.
  • Keep the same testing standard across all team members and locations.

Inspect Mode & DevTools

Always start testing using browser DevTools. All layout issues must be reproducible and debugged in Inspect mode first.

Basic workflow

Required
  • Open DevTools: F12 or right click ‚Üí Inspect.
  • Use the element picker to inspect layout (margin, padding, fonts, colors).
  • Check computed styles when debugging unexpected spacing or font sizes.
  • Use temporary inline changes (in Styles tab) before editing actual CSS.

Responsive mode

Required
  • Toggle device toolbar (Chrome): icon ‚ÄúToggle device toolbar‚Äù or Ctrl+Shift+M/Cmd+Shift+M.
  • Test at minimum width 375px (mobile) and breakpoint 768px (tablet/desktop).
  • Use custom sizes for real devices used by our users (see Devices section).
  • Check scroll behavior, sticky headers, and sliders in both mobile and desktop modes.

Layout debugging tips

Recommended
  • Use temporary outline for debugging: 
    * { outline: 1px solid rgba(90, 200, 250, 0.25); }
  • Check for overflow issues (horizontal scroll) using: 
    html, body { overflow-x: hidden; }
    and then locating the element that causes overflow.
  • Use the ‚ÄúLayers‚Äù and ‚ÄúPerformance‚Äù tabs when debugging complex animations or repaint issues.

Devices & Viewports

We follow a mobile-first approach. Layouts must be tested at defined breakpoints and on real devices when possible.

Target viewports

Required

Use these viewport widths in DevTools ‚ÄúResponsive‚Äù mode as a minimum testing set:

  • 375 √ó 667 ‚Äì baseline mobile (iPhone SE / small Android).
  • 414 √ó 896 ‚Äì large mobile (iPhone 11/12/13/14).
  • 768 √ó 1024 ‚Äì tablet portrait (iPad / small tablets).
  • 1024 √ó 768 ‚Äì tablet landscape.
  • 1280 √ó 720 ‚Äì small desktop / laptop.
  • 1440 √ó 900 ‚Äì standard desktop viewport.

Real device testing

Recommended
  • For every major release, test at least on one iOS and one Android device (latest OS).
  • Pay special attention to touch interactions: sliders, dropdowns, hover states replaced with tap.
  • Test on low/medium network speed (3G/4G) and check initial loading time.
  • Verify that font sizes are readable and hit areas (buttons, links) are large enough for touch.

Browser Support & Coverage

We do not support every browser/version combination. For each project we define a target support policy, and every release must be tested against the minimum coverage below.

Support policy (default)

Required
  • Evergreen browsers: support the latest stable and previous major version (a.k.a. ‚Äúlast 2 major versions‚Äù).
  • iOS Safari: support the current and previous iOS major versions.
  • Android: support Chrome (latest stable and previous major). If the product is embedded in an app, test the relevant WebView versions used by that app.
  • If a project requires legacy support (older iOS, kiosk devices, embedded browsers), the project owner must explicitly document the required versions in the project README and QA checklist.

Desktop (minimum test set)

Required
  • Chrome ‚Äî latest stable + previous major.
  • Edge ‚Äî latest stable + previous major.
  • Firefox ‚Äî latest stable + previous major.
  • Safari (macOS) ‚Äî recommended when the audience is macOS-heavy.

Mobile (minimum test set)

Required
  • Safari (iOS) ‚Äî current + previous iOS major.
  • Chrome (Android) ‚Äî latest stable + previous major.
  • In-app / WebView ‚Äî test if your product is opened inside an app container (e.g., a native app, LINE, etc.).

Explicitly not supported

Required
  • Internet Explorer ‚Äî not supported.
  • Outdated browsers (beyond the policy above) ‚Äî not supported unless the project owner requests it in writing.

Compatibility checks

Recommended
  • For advanced CSS/JS features, verify support in a compatibility database (e.g., ‚ÄúCan I use‚Äù).
  • Prefer progressive enhancement: ship a baseline experience for all supported browsers and enhance when features are available.
  • If your build uses a browser targeting config (e.g. Browserslist), it should match the policy above and is considered the single source of truth.

Testing Checklist (Pre-release)

Before release, the following testing items must be completed and confirmed.

  • ‚úÖ Layout is correct at 375px, 768px, 1024px, and common desktop widths (no horizontal scroll).
  • ‚úÖ All interactive elements (buttons, links, sliders) work correctly on both mouse and touch devices.
  • ‚úÖ No console errors or unhandled JavaScript exceptions.
  • ‚úÖ Sliders (Swiper) work correctly on mobile and desktop, including swiping and navigation.
  • ‚úÖ Critical pages tested in Chrome, Safari (iOS), and Chrome (Android).
  • ‚úÖ Basic accessibility checks: focus states, keyboard navigation, alt texts for images.
  • ‚úÖ Lighthouse run shows acceptable scores (see Performance page).
