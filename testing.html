Testing & QA

Testing Guidelines

This page describes our standard testing workflow: how to use browser DevTools (Inspect mode), test layouts on different devices, and verify cross-browser behavior before release.

Goals

  • Detect layout and responsive issues early (in DevTools).
  • Test on real and emulated devices before release.
  • Keep the same testing standard across all team members and locations.

Inspect Mode & DevTools

Always start testing using browser DevTools. All layout issues must be reproducible and debugged in Inspect mode first.

Basic workflow

Required
  • Open DevTools: F12 or right click ‚Üí Inspect.
  • Use the element picker to inspect layout (margin, padding, fonts, colors).
  • Check computed styles when debugging unexpected spacing or font sizes.
  • Use temporary inline changes (in Styles tab) before editing actual CSS.

Responsive mode

Required
  • Toggle device toolbar (Chrome): icon ‚ÄúToggle device toolbar‚Äù or Ctrl+Shift+M/Cmd+Shift+M.
  • Test at minimum width 375px (mobile) and breakpoint 768px (tablet/desktop).
  • Use custom sizes for real devices used by our users (see Devices section).
  • Check scroll behavior, sticky headers, and sliders in both mobile and desktop modes.

Layout debugging tips

Recommended
  • Use temporary outline for debugging: 
    * { outline: 1px solid rgba(90, 200, 250, 0.25); }
  • Check for overflow issues (horizontal scroll) using: 
    html, body { overflow-x: hidden; }
    and then locating the element that causes overflow.
  • Use the ‚ÄúLayers‚Äù and ‚ÄúPerformance‚Äù tabs when debugging complex animations or repaint issues.

Devices & Viewports

We follow a mobile-first approach. Layouts must be tested at defined breakpoints and on real devices when possible.

Required

  • 414 √ó 896 ‚Äì large mobile (iPhone 11/12/13/14).
  • 768 √ó 1024 ‚Äì tablet portrait (iPad / small tablets).
  • 1024 √ó 768 ‚Äì tablet landscape.
  • 1280 √ó 720 ‚Äì small desktop / laptop.
  • 1440 √ó 900 ‚Äì standard desktop viewport.

Real device testing

Recommended
  • For every major release, test at least on one iOS and one Android device (latest OS).
  • Pay special attention to touch interactions: sliders, dropdowns, hover states replaced with tap.
  • Test on low/medium network speed (3G/4G) and check initial loading time.
  • Verify that font sizes are readable and hit areas (buttons, links) are large enough for touch.

Browser Coverage

We do not support all browsers in the world, but we require a minimal set to be tested before any release.

Desktop

Required
  • Google Chrome (latest stable).
  • Microsoft Edge (latest stable).
  • Firefox (latest stable) for layout sanity checks.

Mobile

Required
  • Safari on iOS (latest stable).
  • Chrome on Android (latest stable).
  • If available, test in WebView containers used by our apps (if applicable).

Notes

Recommended
  • Do not optimize specifically for outdated browsers unless requested by the project owner.
  • For complex features (3D, advanced CSS), always check browser compatibility (Can I Use).
