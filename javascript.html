Overview

This guide covers modern JavaScript best practices for writing clean, maintainable, and scalable code. We follow ES6+ standards and prioritize readability, performance, and developer experience.

Current Default (Today)

  • Language level: ES6+ (modern syntax), browser-first
  • Approach: Vanilla JavaScript with small modules; no framework required
  • DOM hooks: Prefer data-* attributes over styling classes
  • Goal: Keep code readable and component-friendly as the codebase grows

Core Principles

  • Write self-documenting code with meaningful names
  • Prefer const over let, avoid var
  • Prefer ES modules (strict mode by default)
  • Follow single responsibility principle
  • Write pure functions when possible
  • Handle errors gracefully

Variables & Data Types

Declaration

  • Use const by default, let when reassignment is needed
  • Never use var
  • Initialize variables when declaring them
  • Use meaningful, descriptive names

‚úì Good 

const userCount = 42;
let isLoading = true;

const MAX_RETRIES = 3;
const API_BASE_URL = 'https://api.example.com';

function calculateTotal(items) {
  // ...
}

‚úó Bad 

var count = 42;
var flag = true;
let a = 3;
let b = 'test';

function calc(x) {
  // ...
}

Naming Conventions

Type Convention Example
Variables camelCase userCount, isLoading
Constants UPPER_SNAKE_CASE MAX_RETRIES, API_BASE_URL
Functions camelCase getUserData, calculateTotal
Classes PascalCase UserProfile, DataService
Private (convention) _leadingUnderscore _internalMethod, _privateField

Type Checking & Coercion

Use strict equality checks and explicit type conversion:

‚úì Good 

// Strict equality
if (count === 0) { }

// Type checking
if (typeof value === 'string') { }

// Explicit conversion
const num = Number(input);
const str = String(number);
const bool = Boolean(value);

‚úó Bad 

// Loose equality
if (count == 0) { }

// Implicit conversion
const num = +input;
const str = '' + number;

// Wrong type check
if (value != null) { }

Functions

Function Declarations

  • Prefer arrow functions for callbacks and short functions
  • Use function declarations for named functions and hoisting
  • Keep functions small and focused (max 20-30 lines)
  • Avoid side effects when possible

Parameters & Defaults

  • Limit parameters to 3 or fewer (use objects for more)
  • Use default parameters instead of conditional assignments
  • Destructure object parameters for clarity
  • Use rest parameters for variable arguments
// Good parameter patterns
function createUser({ 
  name, 
  email, 
  age = 18, 
  ...otherProps 
}) {
  return { name, email, age, ...otherProps };
}

// Default parameters
function fetchData(url, options = {}) {
  const { timeout = 5000, retries = 3 } = options;
  // ...
}

Error Handling

Always handle errors gracefully in async functions:

// Async/await with try-catch
async function getUserData(userId) {
  try {
    const response = await fetch(`/api/users/${userId}`);
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }
    return await response.json();
  } catch (error) {
    console.error('Failed to fetch user:', error);
    throw new ApiError('USER_FETCH_FAILED', { userId, cause: error });
  }
}

// Custom error classes
class ApiError extends Error {
  constructor(code, details = {}) {
    super(`API Error: ${code}`);
    this.name = 'ApiError';
    this.code = code;
    this.details = details;
  }
}

Async Programming

Promises vs Async/Await

  • Use async/await for most cases (cleaner syntax)
  • Use Promises directly for parallel operations
  • Always handle promise rejections
  • Avoid mixing async/await with .then() chains

Error Handling Patterns

‚úì Good 

// Async/await with try-catch
async function fetchData() {
  try {
    const data = await api.get();
    return process(data);
  } catch (error) {
    handleError(error);
    throw error;
  }
}

// Promise.all with error handling
async function loadAll() {
  try {
    const [users, posts] = await Promise.all([
      fetchUsers(),
      fetchPosts()
    ]);
    return { users, posts };
  } catch (error) {
    // Handle partial failures
  }
}

‚úó Bad 

// Unhandled promise
async function risky() {
  const data = await fetch(); // No error handling
  return data;
}

// Nested promises
fetchUser()
  .then(user => {
    fetchPosts(user.id)
      .then(posts => {
        // Pyramid of doom
      });
  });

Performance Considerations

  • Use Promise.all() for parallel independent operations
  • Use Promise.allSettled() when all promises should complete
  • Implement timeout patterns for long-running operations
  • Consider lazy loading for non-critical async operations
// Parallel execution with timeout
async function fetchWithTimeout(url, timeout = 5000) {
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), timeout);

  try {
    const response = await fetch(url, { signal: controller.signal });
    clearTimeout(timeoutId);
    return response.json();
  } catch (error) {
    clearTimeout(timeoutId);
    throw error;
  }
}

Modules & Imports

ES Modules Syntax

  • Use named exports for utilities and constants
  • Use default exports for single-purpose modules
  • Group imports logically (built-ins, externals, internals)
  • Avoid circular dependencies
// utils/math.js - Named exports
export const PI = 3.14159;

export function sum(a, b) {
  return a + b;
}

export function average(numbers) {
  return sum(...numbers) / numbers.length;
}

// services/api.js - Default export
export default class ApiService {
  // ...
}

// main.js - Import patterns
import { PI, sum } from './utils/math.js';
import ApiService from './services/api.js';
import { debounce } from 'lodash-es';

Tree Shaking & Optimization

Structure modules for optimal bundling:

  • Use named exports to enable tree shaking
  • Avoid side effects in module root
  • Split large modules into focused ones
  • Use dynamic imports for code splitting
// Dynamic import for code splitting
async function loadFeature(featureName) {
  try {
    const module = await import(`./features/${featureName}.js`);
    return module.default;
  } catch (error) {
    console.error(`Failed to load ${featureName}:`, error);
    return null;
  }
}

// Barrel exports (index.js)
export { Button } from './button.js';
export { Input } from './input.js';
export { Modal } from './modal.js';

Objects & Arrays

Modern Syntax Features

  • Use object destructuring for cleaner code
  • Use spread operator for copying and merging
  • Use computed property names for dynamic keys
  • Prefer Object methods over for...in loops

Array Methods

Prefer functional array methods over loops:

‚úì Good (Functional) 

// Transform array
const prices = items
  .filter(item => item.inStock)
  .map(item => item.price * 1.2)
  .reduce((sum, price) => sum + price, 0);

// Find elements
const user = users.find(u => u.id === userId);
const activeUsers = users.filter(u => u.isActive);

// Check conditions
const allValid = values.every(v => v > 0);
const anyValid = values.some(v => v > 0);

‚úó Bad (Imperative) 

// Manual loops
let prices = 0;
for (let i = 0; i < items.length; i++) {
  if (items[i].inStock) {
    prices += items[i].price * 1.2;
  }
}

let user = null;
for (let i = 0; i < users.length; i++) {
  if (users[i].id === userId) {
    user = users[i];
    break;
  }
}

Immutability Patterns

Avoid mutating objects and arrays directly:

// Immutable updates
const updatedUser = {
  ...user,
  profile: {
    ...user.profile,
    email: 'new@email.com'
  }
};

const newArray = [...array, newItem];
const filteredArray = array.filter(item => item.keep);

// Object operations
const { password, ...userWithoutPassword } = user;
const merged = { ...defaults, ...overrides };

// Array operations
const [first, second, ...rest] = array;
const newArray = array.map(item => ({ ...item, updated: true }));

Classes & OOP

Class Structure

  • Use classes when you need inheritance or multiple instances
  • Prefer composition over inheritance
  • Use private fields (#) for true privacy
  • Keep classes focused and cohesive
// Good class example
class UserService {
  #apiClient;
  #cache = new Map();

  constructor(apiClient) {
    this.#apiClient = apiClient;
  }

  async getUser(id) {
    // Check cache first
    if (this.#cache.has(id)) {
      return this.#cache.get(id);
    }

    // Fetch from API
    const user = await this.#apiClient.fetchUser(id);
    this.#cache.set(id, user);
    return user;
  }

  clearCache() {
    this.#cache.clear();
  }
}

// Usage
const service = new UserService(apiClient);
const user = await service.getUser(123);

Inheritance vs Composition

‚úì Composition 

class Logger {
  log(message) {
    console.log(`[LOG] ${message}`);
  }
}

class ApiService {
  constructor(logger) {
    this.logger = logger;
  }

  async fetch() {
    this.logger.log('Fetching data...');
    // ...
  }
}

const service = new ApiService(new Logger());

‚úó Deep Inheritance 

class Animal {}
class Mammal extends Animal {}
class Dog extends Mammal {}
class ServiceDog extends Dog {}
// Fragile base class problem

Libraries & Dependencies

Philosophy: Build vs Buy

Every dependency is a trade-off. Consider these questions:

Build Your Own

  • Simple, focused functionality needed
  • Custom behavior specific to your app
  • Learning opportunity for the team
  • Bundle size is critical
  • Long-term maintenance control

Use a Library

  • Complex problem (dates, charts, animations)
  • Industry-standard solution exists
  • Accessibility requirements
  • Cross-browser compatibility needed
  • Time-to-market is critical

Decision Framework

IF (problem is complex) AND 
    (library is well-maintained) AND 
    (bundle size < 50KB gzipped) AND
    (solves multiple use cases) THEN
  USE library
ELSE IF (functionality is simple) OR
    (custom behavior needed) OR
    (bundle size critical) OR
    (learning opportunity) THEN
  BUILD custom
END

jQuery: Legacy Context

‚ö†Ô∏è jQuery in Modern Projects

Avoid jQuery in new projects. Most jQuery functionality is now available natively:

jQuery Pattern
$('.button').on('click', () => {
  $(this).toggleClass('active');
  $.ajax({ url: '/api' });
});

$('.items').each(function() {
  $(this).find('.child');
});
Modern Native Equivalent
// DOM selection
document.querySelectorAll('.button')
  .forEach(btn => {
    btn.addEventListener('click', (e) => {
      e.target.classList.toggle('active');
      fetch('/api');
    });
  });

// Iteration
document.querySelectorAll('.items')
  .forEach(item => {
    item.querySelector('.child');
  });

When jQuery Might Still Make Sense

  • Maintaining legacy codebase (migration in progress)
  • Prototyping quickly (though modern alternatives exist)
  • Specific plugins with no modern equivalent

Migration path: Use cash-dom or umbrella-js as lighter jQuery alternatives if needed.

Modern Library Principles

Tree-shakable

Essential

Import only what you use. Prefer ES modules over CommonJS.

Types / Documentation

Preferred

Prefer libraries with clear docs and stable APIs. If types are available, use them; TypeScript adoption can be phased in later.

Framework Agnostic

Preferred

Works with any framework (React, Vue, Svelte) or vanilla JS.

Zero Dependencies

Preferred

Minimal external dependencies reduces vulnerability surface.

When to Use Splide:
  • Complex carousels with multiple slides visible
  • Touch/swipe gestures required on mobile
  • Lazy loading of images in slider
  • Autoplay with pause on hover/interaction
  • Thumbnail navigation synchronized with main slider
What Splide Adds:

// Features you get:
- Touch/swipe gestures
- Keyboard navigation
- Accessibility (ARIA labels)
- Responsive breakpoints
- Lazy loading
- Autoplay
- Pagination
- Navigation arrows
- Thumbnails sync
‚úì Good Use Case 
// Product image gallery
import Splide from '@splidejs/splide';

const gallery = new Splide('#gallery', {
  type       : 'fade',
  rewind     : true,
  pagination : false,
  arrows     : true,

const thumbnails = new Splide('#thumbnails', {

‚úó Unnecessary Use 
// Simple image slideshow

const slider = new Splide('.slider', {
  type      : 'slide',
  perPage   : 1,
  autoplay  : true,
  interval  : 3000,
});

slider.mount();

// Could be replaced with:
function simpleSlider(selector) {
  const slides = document.querySelectorAll(selector);
  let current = 0;
  
  setInterval(() => {
    slides[current].classList.remove('active');
    current = (current + 1) % slides.length;
    slides[current].classList.add('active');
  }, 3000);
}

Overkill: 22KB for basic auto-advance

üö® Common Pitfalls with Slider Libraries:
  • Performance: Initializing on many elements
  • CLS (Cumulative Layout Shift): Slider height not reserved
  • Accessibility: Forgetting keyboard nav/ARIA
  • Mobile: Touch conflicts with page scroll
  • SEO: Content in sliders not indexed properly

üéØ When to Choose UI Libraries:
  • Yes: Complex widgets (date pickers, rich text editors)
  • Yes: Accessibility-critical components (modals, tooltips)
  • Maybe: Common components (buttons, inputs) if design system needed
  • No: Simple components (accordion, tabs) that are easy to build

‚ö†Ô∏è Library-Specific Anti-Patterns

Splide/Swiper Anti-Patterns:

  • Initializing on hidden elements - causes layout shifts
  • Multiple instances - one per page is usually enough
  • Forgetting destroy() - memory leaks in SPA
  • Over-configuring - use defaults when possible
  • Ignoring SSR/hydration ‚Äî can break when markup is server-rendered and JavaScript enhances it

Fix Strategies:

  • Lazy initialize: Only when element is visible
  • Shared config: Reuse configuration objects
  • Cleanup: Always call destroy() in your component teardown/unmount
  • Conditional loading: Dynamic import for heavy libraries
  • Monitoring: Track bundle size impact in CI 
// GOOD: Conditional Splide loading
async function loadSliderIfNeeded() {
  const element = document.getElementById('product-gallery');
  if (!element || window.innerWidth < 768) return;
  
  // Only load on desktop for complex gallery
  const Splide = await import('@splidejs/splide');
  
  new Splide.default(element, {
    // minimal config
    perPage: 3,
    gap: '20px',
  }).mount();
}

// BAD: Always loading
import Splide from '@splidejs/splide'; // 22KB always

Library Performance Budget

Set and enforce limits for library usage:

Green Zone (< 50KB)

  • Essential utilities (date-fns, axios)
  • Critical UI (modals, tooltips)
  • Performance helpers (debounce, throttle)
  • No approval needed

Yellow Zone (50-150KB)

  • Complex UI (Splide, chart libraries)
  • State management (Zustand, Jotai)
  • Form handling libraries
  • Team lead approval required

Red Zone (> 150KB)

  • Full UI frameworks (Material-UI)
  • Monolithic libraries (full Lodash)
  • Heavy visualization (D3, Three.js)
  • Architecture review required

Error Handling

Custom Error Classes

Create specific error types for better handling:

// Base application error
class AppError extends Error {
  constructor(message, code = 'APP_ERROR', details = {}) {
    super(message);
    this.name = this.constructor.name;
    this.code = code;
    this.details = details;
    this.timestamp = new Date().toISOString();
    
    // Capture stack trace
    Error.captureStackTrace?.(this, this.constructor);
  }
}

// Specific error types
class ValidationError extends AppError {
  constructor(field, message) {
    super(`Validation failed: ${message}`, 'VALIDATION_ERROR', { field });
  }
}

class NetworkError extends AppError {
  constructor(url, status) {
    super(`Network request failed`, 'NETWORK_ERROR', { url, status });
  }
}

// Usage
try {
  throw new ValidationError('email', 'Invalid email format');
} catch (error) {
  if (error instanceof ValidationError) {
    console.error(`Field ${error.details.field}: ${error.message}`);
  }
}

Global Error Handling

// Window error handler
window.addEventListener('error', (event) => {
  console.error('Global error:', event.error);
  logToService(event.error);
  // Prevent default browser error display if needed
  event.preventDefault();
});

// Unhandled promise rejection
window.addEventListener('unhandledrejection', (event) => {
  console.error('Unhandled promise rejection:', event.reason);
  logToService(event.reason);
  event.preventDefault();
});

// Async error boundary
async function withErrorBoundary(operation, fallback) {
  try {
    return await operation();
  } catch (error) {
    console.error('Operation failed:', error);
    return fallback?.(error) ?? null;
  }
}

Security

XSS Prevention

  • Always escape user input before rendering
  • Use Content Security Policy (CSP)
  • Sanitize HTML with DOMPurify or similar
  • Avoid innerHTML with user content
// Safe HTML rendering
function renderUserContent(text) {
  const element = document.createElement('div');
  element.textContent = text; // Automatically escaped
  return element;
}

// With DOMPurify
import DOMPurify from 'dompurify';

function renderRichContent(html) {
  const clean = DOMPurify.sanitize(html, {
    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a'],
    ALLOWED_ATTR: ['href', 'title']
  });
  element.innerHTML = clean;
}

// URL validation
function isValidUrl(string) {
  try {
    const url = new URL(string);
    return ['http:', 'https:'].includes(url.protocol);
  } catch {
    return false;
  }
}

Vite (Future Direction)

Why Consider Vite

Vite is a modern frontend build tool that improves the developer experience for JavaScript modules and styles (SCSS/PostCSS), while producing optimized production bundles. It does not require adopting a framework.

  • Faster iteration: modern dev server and incremental updates
  • Cleaner module workflow: native ES module imports for component scripts
  • Production build: consistent bundling, minification, and cache-friendly asset output
  • Scales with complexity: easier to keep JS/CSS organized as a component library grows

Trade-offs

  • Node toolchain: adds dependency management (npm) and build config to the repo
  • Team onboarding: requires consistent scripts and version discipline
  • Migration effort: best introduced on new work first, not as a forced rewrite

How This Affects Our Current JavaScript Practices

  • These guidelines still apply: naming, immutability, error handling, and modular structure.
  • ES modules become the default for component scripts (imports/exports instead of global scripts).
  • Bundle/performance budgets remain relevant; the measurement tools may change (e.g., Rollup/Vite visualizers).

Adoption Path (Low Risk)

  1. Pilot on one small project with SCSS + modular JS.
  2. Create a starter template (scripts, folder conventions, linting).
  3. Use it for new work; migrate older projects only when the ROI is clear.

Async Patterns & Examples

Promise Patterns

// Creating promises
function delay(ms) {
  return new Promise((resolve) => {
    setTimeout(resolve, ms);
  });
}

function fetchWithRetry(url, retries = 3) {
  return new Promise((resolve, reject) => {
    const attempt = (n) => {
      fetch(url)
        .then(resolve)
        .catch(error => {
          if (n > 0) {
            console.log(`Retrying... (${n} attempts left)`);
            attempt(n - 1);
          } else {
            reject(error);
          }
        });
    };
    attempt(retries);
  });
}

// Promise utilities
const timeout = (promise, ms) => {
  return Promise.race([
    promise,
    new Promise((_, reject) => {
      setTimeout(() => reject(new Error('Timeout')), ms);
    })
  ]);
};

Async/Await Patterns

// Sequential vs parallel execution
async function sequentialFetch() {
  const user = await fetchUser();
  const posts = await fetchPosts(user.id); // Waits for user first
  const comments = await fetchComments(posts[0].id);
  return { user, posts, comments };

async function parallelFetch() {
  const [user, posts, comments] = await Promise.all([
    fetchUser(),
    fetchPosts(),
    fetchComments()

  return { user, posts, comments }; // All fetched in parallel
}

Real-world Async Patterns

// Rate limiting with queues
class RateLimiter {
  constructor(requestsPerSecond) {
    this.queue = [];
    this.interval = 1000 / requestsPerSecond;
    this.processing = false;
  }

  async add(request) {
    return new Promise((resolve) => {
      this.queue.push({ request, resolve });
      if (!this.processing) this.process();
    });
  }

  async process() {
    this.processing = true;
    while (this.queue.length > 0) {
      const { request, resolve } = this.queue.shift();
      const result = await request();
      resolve(result);
      await delay(this.interval);
    }
    this.processing = false;
  }
}

// Concurrent request management
async function fetchWithConcurrency(urls, maxConcurrent = 5) {
  const results = [];
  const executing = new Set();
  
  for (const url of urls) {
    const promise = fetch(url).then(res => res.json());
    results.push(promise);
    executing.add(promise);
    
    promise.then(() => executing.delete(promise));
    
    if (executing.size >= maxConcurrent) {
      await Promise.race(executing);
    }
  }
  
  return Promise.all(results);
}

Performance Optimization Patterns

Debouncing & Throttling

// Debounce implementation
function debounce(func, wait) {
  let timeout;
  return function executedFunction(...args) {
    const later = () => {
      clearTimeout(timeout);
      func(...args);
    };
    clearTimeout(timeout);
    timeout = setTimeout(later, wait);
  };
}

// Throttle implementation
function throttle(func, limit) {
  let inThrottle;
  return function(...args) {
    if (!inThrottle) {
      func.apply(this, args);
      inThrottle = true;
      setTimeout(() => inThrottle = false, limit);
    }
  };
}

// Usage
const handleResize = debounce(() => {
  console.log('Window resized');
  updateLayout();
}, 250);

window.addEventListener('resize', handleResize);

const handleScroll = throttle(() => {
  console.log('Scrolling');
  checkPosition();
}, 100);

window.addEventListener('scroll', handleScroll);

Memoization & Caching

// Simple memoization
function memoize(fn) {
  const cache = new Map();
  return function(...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) {
      return cache.get(key);
    }
    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

// Expensive calculation
function fibonacci(n) {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}

const memoizedFib = memoize(fibonacci);

// Usage with cache clearing
function createCachedApiCall(apiFunc) {
  const cache = new Map();
  const maxSize = 100;
  
  return async function(key) {
    if (cache.has(key)) {
      console.log('Cache hit:', key);
      return cache.get(key);
    }
    
    console.log('Cache miss:', key);
    const result = await apiFunc(key);
    
    // LRU cache eviction
    if (cache.size >= maxSize) {
      const firstKey = cache.keys().next().value;
      cache.delete(firstKey);
    }
    
    cache.set(key, result);
    return result;
  };
}

Lazy Loading & Code Splitting

// Lazy loading images
function lazyLoadImages() {
  const images = document.querySelectorAll('img[data-src]');
  const observer = new IntersectionObserver((entries) => {
    entries.forEach(entry => {
      if (entry.isIntersecting) {
        const img = entry.target;
        img.src = img.dataset.src;
        img.removeAttribute('data-src');
        observer.unobserve(img);
      }
    });
  }, { rootMargin: '50px' });

  images.forEach(img => observer.observe(img));
}

date-fns ‚úì
  • Modular (tree-shakable)
  • Immutable
  • Small bundle (~300KB total)
  • Active maintenance
  • TypeScript native
moment.js ‚úó
  • Monolithic (~4MB minified)
  • Mutable
  • Large bundle size
  • Legacy mode
