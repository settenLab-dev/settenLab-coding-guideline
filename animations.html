Introduction

This website defines the official frontend guidelines for our company. It is used by all frontend developers in Japan and Uzbekistan to keep HTML, CSS, and JavaScript consistent across projects.

Goals

  • Ensure consistent code style and structure.
  • Make onboarding new developers easier.
  • Share reusable patterns for components, sliders, and animations.

Attribute Naming Guidelines

Attributes define behavior, state, and accessibility. Class names are used primarily for styling, while data-* and ARIA attributes are used for JavaScript hooks, state, analytics, and a11y. This section defines our standard patterns.

Purpose Format Example Notes
JS hook data-js-* data-js-modal-open="contact"
data-js-slider="hero" 		Used only for JavaScript bindings. Never rely on classes for JS behavior.
Component ID data-component data-component="slider" Identifies a component root in DOM for initialization and debugging.
Component role data-role data-role="track",
data-role="slide",
data-role="close" 		Internal sub-elements inside a component. Not used for styling.
State data-state data-state="open",
data-state="closed",
data-state="loading",
data-state="error" 		Current state of the component. Must be in sync with ARIA attributes.
Variant data-variant data-variant="hero",
data-variant="compact" 		Describes a functional variation (type) of component instance.
Analytics data-track-* data-track-click="add_to_cart",
data-track-view="hero_banner" 		Used by tracking/analytics only. Must not affect behavior or styling.
API / data binding data-api-* data-api-id="123",
data-api-endpoint="/products/123" 		References to back-end resources or data identifiers used in JS.
Accessibility ARIA attributes aria-expanded="false",
aria-hidden="true",
aria-controls="menu-panel" 		Used strictly for accessibility. Always kept consistent with data-state.

Rule: classes (.class-name) define visuals. Attributes (data-*, ARIA) define behavior, state, and semantics. Do not mix styling and behavior in the same construct.

Units & Sizing

This guide defines when to use rem, em, and px. It follows modern CSS architecture practices used in design systems (Material, Polaris, Primer). Mobile-first, scalable, and predictable sizing rules improve consistency and maintainability.

rem (Root-relative)

Primary unit for typography, spacing, component sizing, layout. Independent from nesting and predictable across the entire UI. Recommended for 90% of sizing.

  • Use for: font-size, padding, margin, gap, layout widths
  • 1rem = root font-size (default 16px)
  • Responsive scaling via root font-size
Open rem preview

em (Element-relative)

Scales with the font-size of the current element. Useful for internal component scaling but can cause unexpected growth if nested.

  • Use inside components (padding, icons)
  • 1em = computed font-size of the element
  • Avoid in layout and spacing outside components
Open em preview

px (Absolute)

Pixel-perfect unit for borders, shadows, and breakpoints. Use where precision overrides scaling.

  • Use for: borders, 1px lines, shadows
  • Safe for breakpoints
  • Not scalable
Open px preview

Warning: Using vw units

Viewport width units (vw) can cause severe layout issues on mobile devices, especially iOS Safari and Chrome Android. Avoid using vw for typography and container widths unless you explicitly control scaling.

  • Text becomes unreadable on phones when using vw
  • iOS hides browser UI ‚Üí viewport jumps ‚Üí layout shifts
  • 100vw includes scrollbar ‚Üí horizontal overflow bugs
  • Use rem for text, % for layout width
  • vw is acceptable only for: fullscreen hero blocks, animations, clipping
Open vw warning preview

Warning: vh, svh, lvh, dvh

Viewport height units can behave inconsistently on mobile devices. Historically, 100vh caused layout jumps because mobile browser UI (address bar) resizes the viewport. Modern units (dvh, svh, lvh) resolve this, but must be used intentionally.

  • vh is unstable on mobile (iOS & Android)
  • 100vh often cuts content or creates extra space
  • dvh is usually the correct modern replacement
  • svh and lvh provide UI-safe variants
  • Use for fullscreen sections only ‚Äî never for main layout height
Open vh / dvh preview

% (Percent)

Percent units define size relative to the parent element. They are ideal for fluid layouts, flexible widths, and responsive components. Use % whenever an element must scale with its container instead of the viewport.

  • Best for fluid width layouts (e.g., width: 100%)
  • Useful in grids, images, media containers
  • Avoid using % for height unless parent has a defined height
  • Combine with max-width for controlled scaling
Open % usage preview

vw ‚Äì Use With Caution

‚úï

vw units behave unpredictably on mobile: browser UI bars resize the viewport, and 100vw often includes scrollbar width. This preview demonstrates typical failure cases and correct alternatives.

Bad: Text sizing with vw

Text 5vw

On small screens 5vw becomes too small, on large screens too big. Also affected by mobile UI shrinking/expanding the viewport.

Bad: 100vw container

100vw box ‚Üí scrollbar causes overflow

Because 100vw includes scrollbar width, layouts get horizontal scroll even when content fits visually.

Good: Hero section using vw/vh

Hero block: width:100%; height:100vh

vw/vh may be used for fullscreen hero blocks, background shapes, animations, clipping masks ‚Äî elements not tied to content readability. 

/* Bad: text with vw */
.title {
  font-size: 5vw; /* unpredictable */
}

/* Bad: layout container */
.container {
  width: 100vw; /* causes overflow */
}

/* Good: hero section */
.hero {
  min-height: 100vh;
  width: 100%;
}

vh vs dvh ‚Äì Modern Viewport Units

‚úï

The classic 100vh unit is unreliable on mobile because the browser UI bars change the viewport height. Modern CSS provides dvh, svh, and lvh to fix this. Avoid using 100vh on mobile. Use 100dvh for fullscreen, stable height. svh = viewport with browser UI visible. lvh = viewport with UI hidden (expanded). dvh = dynamic, reliable ‚Äî recommended default.

Bad: 100vh (mobile jump)

100vh

On iOS Safari, 100vh becomes higher when the address bar hides, causing layout jumps and clipped content.

Good: 100dvh (stable viewport)

100dvh

dvh uses the dynamic viewport height: always accurate, stable, and recommended for fullscreen blocks.

svh / lvh

visual viewport variants

svh = small viewport height (browser UI visible)
lvh = large viewport height (UI hidden)
Rarely needed ‚Äî mostly for advanced layouts. 

/* Bad */
.hero {
  height: 100vh; /* unstable on mobile */
}

/* Recommended */
.hero {
  height: 100dvh; /* dynamic viewport height */
}

/* Optional advanced usage */
.hero--safe {
  min-height: 100svh; /* UI visible */
  max-height: 100lvh; /* UI hidden */
}

% ‚Äì Relative to Parent

‚úï

Percent units adapt an element‚Äôs width or position based on its parent. They are essential for fluid layout systems and responsive containers. The preview below demonstrates correct and incorrect use cases.

Good: width: 100%

Child expands exactly to the available space of the parent. Perfect for fluid layouts, responsive grids, and images.

Bad: height: 100%

height:100% works only if the parent has an explicit height. Otherwise it collapses or produces unpredictable results.

Good: max-width + %

Combine % with max-width for controlled fluidity. Ideal for containers and responsive media. 

/* Fluid width */
.container {
  width: 100%;
}

/* Bad: parent needs explicit height */
.section {
  height: 100%; /* Only works if parent has height set */
}

/* Fluid but controlled */
.media {
  width: 100%;
  max-width: 600px;
}

Sliders & Carousels

We use a unified approach for sliders (e.g. Swiper, Splide). This section should explain how to initialize them and what options are allowed.

General Rules

  • Use one official library (e.g. Swiper) across all projects.
  • Initialization code must be in a separate JS module or clearly named function.
  • Provide keyboard navigation and focus states.

Example: Swiper Initialization (pseudo)

const slider = new Swiper('.js-main-slider', {
  loop: true,
  speed: 600,
  slidesPerView: 1,
  navigation: {
    nextEl: '.js-main-slider-next',
    prevEl: '.js-main-slider-prev',
  },
  pagination: {
    el: '.js-main-slider-pagination',
    clickable: true,
  },
});

Animations

Animations must support performance and not distract users. Prefer CSS transitions and transforms. Heavy animations should be optional.

Rules

  • Use transform and opacity for smooth animations.
  • Max duration for micro-interactions: 150‚Äì250ms.
  • Respect prefers-reduced-motion setting.

Example: Button Hover

.btn {
  transition: transform 0.18s ease-out, box-shadow 0.18s ease-out;
}

.btn:hover {
  transform: translateY(-1px);
  box-shadow: 0 8px 18px rgba(0, 0, 0, 0.25);
}

Best Practices

Do

  • Write self-explanatory class names.
  • Keep components small and reusable.
  • Document any non-standard behavior.

Don't

  • Don‚Äôt inline large chunks of CSS or JS into HTML.
  • Don‚Äôt use different libraries for the same task in one project.
  • Don‚Äôt override guidelines without discussion.
