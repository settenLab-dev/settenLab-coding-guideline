Components

Overview

Component-based architecture helps us build scalable, maintainable, and reusable user interfaces. This guide documents how we structure components today (HTML + SCSS with BEM + JavaScript) and also includes optional patterns (Web Components, frameworks) for cases where they are justified.

Current default (today)

  • Markup: server-rendered HTML (partials/includes).
  • Styling: SCSS with BEM naming and a small set of shared tokens (CSS variables).
  • Behavior: modular JavaScript, initialized via data-* hooks (no JS selectors based on BEM classes).
  • Component mindset: reusable sections/blocks with a local folder + README + examples.
  • Optional reference: Web Components and React/TypeScript are included for future or special-case needs, not as a default requirement.

Core Principles

  • Single responsibility: A component does one thing and does it well.
  • Explicit API: Clear contract for what can be customized (variants/modifiers) and what cannot.
  • Isolation: Component styles and scripts don't break neighboring elements.
  • Documentability: Usage examples and variant list for each component.

Choosing the Right Approach

Most projects should start with the baseline approach: HTML/CSS/JS components with BEM styling and data-* hooks for behavior. Consider Web Components when you need stronger encapsulation. Consider a framework only when you are building an app-like interface with substantial client-side state.

BEM as Core Contract

BEM (Block‚ÄëElement‚ÄëModifier) provides a formal naming convention that turns component thinking into a clear contract: Block = component, Element = component part, Modifier = variant (state/size/appearance).

BEM Rules

  • Block: Stand‚Äëalone component: .button, .card, .hero
  • Element: Part of a block: .card__title, .card__meta
  • Modifier: Variant of block/element: .button--primary, .button--sm, .is‚Äëdisabled (by convention)
  • JS hooks: Use data‚Äëjs/data‚Äëcomponent instead of "js‚Äë‚Ä¶" classes to separate behavior from styling.

JavaScript & Contracts

For "simple" websites, the most resilient pattern is initialization via data‚Äëattributes. This separates styling (BEM) from behavior and reduces regression risk.

Pattern: data‚Äëcomponent + module

<div class="accordion" data-component="accordion">...</div>
// accordion.js
export function initAccordion(root) {
  // root ‚Äî specific component container
}

export function initAllAccordions() {
  document
    .querySelectorAll('[data-component="accordion"]')
    .forEach(initAccordion);
}

Conventions

  • Avoid "magic" selectors with long chains (.page .content ...).
  • Emit events outward: CustomEvent for cross‚Äëcomponent integration.
  • States (open/active/disabled) via BEM modifiers or data‚Äëstate attributes.

Consistent Styling: Tokens & Variables

To keep components visually cohesive, we need a shared layer of "design tokens." At a basic level, these can be CSS custom properties used across all blocks.

Minimal Token Set (Example)

:root {
  --color-text: #111;
  --color-bg: #fff;
  --color-primary: #2b6cb0;

  --radius-md: 12px;
  --space-2: 8px;
  --space-3: 12px;
  --space-4: 16px;
}
  • Rule: Components don't "know" specific colors or spacings‚Äîonly tokens.
  • Theming: If brand/theme changes, update tokens, not rewrite components.

Quality: A11y, Responsiveness, Performance

Accessibility (Minimum)

  • Focus styles must be visible (never remove outline without a replacement).
  • Interactive elements use semantic tags (button, a), not div.
  • Modals/menus: keyboard navigation, proper aria-* attributes.

Responsiveness

  • Components must be resilient to varying text length and localization.
  • Check layouts at 320px and on large screens.

Performance

  • Avoid JavaScript where CSS suffices (e.g., simple expandable blocks).
  • Lazy‚Äëinitialize heavy components (on click/viewport).
  • Images: loading="lazy", proper sizes.

When to Use a Framework

A framework is not a goal by itself. Start by strengthening the baseline approach (component catalog, tokens, consistent conventions, and modular JS). Consider a framework only when it materially reduces complexity or risk‚Äîand prefer adopting it in isolated areas first, rather than rewriting everything.

Symptom What's Happening What to Do
Complex UI State Many interdependent states, growing number of "flags" Consider component framework or Web Components for "islands"
UI Duplication Identical blocks are copied and diverge Start with catalog + tokens; at scale‚Äîcomponent library
Team Growing Harder to align changes and standards Standardize the component catalog and conventions, add documentation/linting

Example: Button in React + TypeScript

type ButtonVariant = "primary" | "secondary";
type ButtonSize = "sm" | "md" | "lg";

export interface ButtonProps {
  variant?: ButtonVariant;
  size?: ButtonSize;
  disabled?: boolean;
  onClick?: () => void;
  children: React.ReactNode;
}

export function Button({
  variant = "primary",
  size = "md",
  disabled,
  onClick,
  children,
}: ButtonProps) {
  return (
    <button
      className={`button button--${variant} button--${size}`}
      disabled={disabled}
      onClick={onClick}
      type="button"
    >
      {children}
    </button>
  );
}

Practices to Aim For

  • Typed API: Variants and sizes as union types; avoid "stringly typed".
  • Documentation: Storybook (or similar) as single catalog.
  • Tokens: Same CSS variables/design tokens for consistent UI.
  • Tests: Unit + e2e for critical user flows.

Web Components Overview

Framework‚Äëagnostic custom elements that work anywhere. Ideal for shared UI libraries, design systems, or embedding components in multiple tech stacks.

When to Use Web Components

  • Share components across different frameworks (React, Vue, etc.)
  • Build a design system that works in any environment
  • Need encapsulated styles and behavior without a framework
  • Progressive enhancement for complex UI elements
View Web Component example ‚Üí

Key Benefits

  • Framework‚Äëagnostic: Works with React, Vue, Angular, or plain HTML
  • Encapsulation: Shadow DOM provides style and DOM isolation
  • Native browser support: No build step required for basic usage
  • Reusability: Same component across multiple projects

Implementation Checklist

Basic (Suitable for current projects)

  • Defined block (BEM) and list of modifiers (variant/size/state).
  • Markup example and constraints documented in README or catalog.
  • Styles don't use global selectors and aren't page‚Äëdependent.
  • JavaScript (if needed) initializes via data‚Äëcomponent.
  • States checked: hover/focus/disabled, text overflow, mobile view.

Advanced (What to aim for)

  • Using design tokens (CSS variables) instead of hard‚Äëcoded values.
  • Minimum a11y checks (keyboard, aria where needed).
  • Tests for critical flows (especially forms/modals/navigation).
  • Component catalog that developers actually use.

Vite as a Future Direction

Vite is not a framework and does not change our BEM or component principles. It is a build tool that can standardize how we develop and ship SCSS/JS across projects. Today we can work with lightweight tooling (e.g., Live Server + SCSS compilation); Vite becomes valuable when we want the build to be reproducible, consistent, and production-oriented by default.

Potential benefits

  • Project-standard build: the build process lives in the repository (not in individual IDE plugins).
  • Production-ready output: optimized assets (minified CSS/JS) and safer caching through hashed filenames.
  • Better modularity: encourages splitting JS into modules and keeping component assets close to the component.
  • Clear dev vs. build modes: consistent commands and predictable behavior across machines.

Costs and trade-offs

  • Node/npm upkeep: versions and dependencies need light maintenance.
  • Small learning curve: developers must adopt dev/build commands and repo-based configuration.
  • Configuration choices: we will need to agree on folder layout and how assets are referenced in templates.

Impact on the current approach

  • No change to BEM: BEM remains the CSS naming contract.
  • Component structure becomes easier: each component can keep its SCSS/JS locally while Vite produces a clean build.
  • Better consistency: fewer ‚Äúworks on my machine‚Äù differences because build rules are shared.

Low-risk adoption path

  1. Pilot Vite on one small project and document the workflow.
  2. Extract a simple starter template (folder structure + commands).
  3. Use it for new projects first; avoid mandatory migration of existing projects.

JavaScript Initialization Pattern

// card.js
export function initCard(cardElement) {
  const buttons = cardElement.querySelectorAll('.button');
  
  buttons.forEach(button => {
    button.addEventListener('click', (e) => {
      e.preventDefault();
      cardElement.classList.toggle('card--active');
      
      // Emit custom event
      cardElement.dispatchEvent(new CustomEvent('card-toggle', {
        bubbles: true,
        detail: { element: cardElement }
      }));
    });
  });
}

// Initialize all cards on page
export function initAllCards() {
  document.querySelectorAll('[data-component="card"]')
    .forEach(initCard);
}

// Auto-initialize if script loaded directly
if (document.readyState === 'loading') {
  document.addEventListener('DOMContentLoaded', initAllCards);
} else {
  initAllCards();
}

Web Component Example

Custom Element Implementation

// user-card.js
class UserCard extends HTMLElement {
  static observedAttributes = ['name', 'avatar', 'role'];
  
  constructor() {
    super();
    this.attachShadow({ mode: 'open' });
  }
  
  connectedCallback() {
    this.render();
    this.setupEvents();
  }
  
  attributeChangedCallback(name, oldValue, newValue) {
    if (oldValue !== newValue) {
      this.render();
    }
  }
  
  render() {
    const name = this.getAttribute('name') || 'Anonymous';
    const avatar = this.getAttribute('avatar');
    const role = this.getAttribute('role') || 'User';
    
    this.shadowRoot.innerHTML = `
      <style>
        :host {
          display: block;
          font-family: system-ui;
          padding: 1rem;
          border: 1px solid #e5e7eb;
          border-radius: 0.5rem;
        }
        
        .avatar {
          width: 64px;
          height: 64px;
          border-radius: 50%;
          margin-bottom: 1rem;
        }
        
        .name {
          font-size: 1.25rem;
          font-weight: 600;
          margin: 0 0 0.25rem 0;
        }
        
        .role {
          color: #6b7280;
          margin: 0;
        }
      </style>
      
      <div class="card">
        ${avatar ? `<img src="${avatar}" alt="${name}" class="avatar">` : ''}
        <h3 class="name">${name}</h3>
        <p class="role">${role}</p>
        <slot></slot>
      </div>
    `;
  }
  
  setupEvents() {
    this.shadowRoot.querySelector('.card')
      .addEventListener('click', () => {
        this.dispatchEvent(new CustomEvent('user-card-click', {
          detail: { name: this.getAttribute('name') },
          bubbles: true
        }));
      });
  }
}

// Register the custom element
customElements.define('user-card', UserCard);
