Introduction

This guideline defines how settenLab builds and maintains front-end code across projects: from HTML and CSS standards, BEM naming and components, to performance, testing, and development environment setup. The goal is to keep our codebase predictable, maintainable, and easy to scale across teams and locations.

When to use this guide

  • Starting a new project or landing page.
  • Refactoring an existing layout or component.
  • Onboarding new developers into the team.
  • Reviewing pull requests for HTML/CSS/JS quality.

Core Principles

All rules in this guideline are built on a few core principles. These should be respected even if a specific rule is not covered explicitly.

01. Mobile-first

Layouts follow a mobile-first approach. Responsive enhancements for larger screens are applied via min-width media queries, and breakpoints are introduced when the layout requires it. There are no desktop-only layouts.

02. BEM & semantics

We use semantic HTML and BEM for class naming. Behavior is driven by data-* attributes, not CSS classes.

03. Components first

UI is built from reusable components: buttons, cards, sliders, media blocks, forms. No one-off styling for isolated pages.

04. Performance by default

Images, fonts, and scripts follow performance rules by default. Lighthouse performance and CLS are first-class metrics.

05. Predictable structure

Projects aim to follow a consistent file structure and naming conventions where applicable (e.g., assets/, components/, pages/).

06. Tooling alignment

We provide recommended VS Code settings, formatters, and linters to align rules locally and in CI where possible, while allowing flexibility in tools and workflows. CI enforces only the agreed baseline checks.
