Markup

Overview

This guideline defines how we write HTML for clarity, accessibility, and long-term maintainability. It favors semantic structure, predictable naming, and progressive enhancement.

Core Principles

  • Semantics first: choose elements for meaning, not appearance
  • Accessibility by default: labels, focus, landmarks, and headings are non-negotiable
  • SEO-ready head: correct metadata, titles, and social previews
  • Progressive enhancement: core UX works without JavaScript
  • Predictable structure: consistency across pages and components

Document Skeleton

Use HTML5 with a minimal, standards-based head. Prefer evergreen browser support and avoid legacy hacks.

Baseline Template

<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">

    <title>Page title ‚Äî Site name</title>
    <meta name="description" content="One-sentence page summary.">

    <link rel="icon" href="/favicon.svg" type="image/svg+xml">
    <link rel="stylesheet" href="./assets/css/style.css">
  </head>
  <body>
    <a class="skip-link" href="#main-content">Skip to content</a>

    <header>...</header>
    <main id="main-content">...</main>
    <footer>...</footer>

    <script type="module" src="./assets/js/main.js"></script>
  </body>
</html>

Encoding, line breaks, indentation

  • UTF-8 without BOM; keep files consistent across the repo
  • LF line endings (configure via .editorconfig)
  • Indent with spaces (2 or 4) consistently; do not mix tabs and spaces within a codebase
  • Never leave trailing whitespace

Note: older internal documents used TAB indentation; modern teams typically standardize on spaces to align with tooling (Prettier/ESLint).

Head & Metadata

Metadata impacts SEO, social previews, and usability. Keep it correct, specific, and page-unique.

Title & description
  • <title>: concise, unique per page. Prefer: Page title ‚Äî Site name
  • description: 1‚Äì2 sentences describing the page content; keep it human-readable
  • Do not keyword-stuff; write for humans first
<title>Markup ‚Äî Coding Guideline</title>
<meta name="description" content="Modern HTML markup rules: semantics, accessibility, SEO metadata, forms, and media patterns.">
Open Graph & social cards

Add social tags for public pages. Keep the image optimized and absolute URL in production. 

<meta property="og:title" content="Markup ‚Äî Coding Guideline">
<meta property="og:description" content="Modern HTML markup rules...">
<meta property="og:type" content="website">
<meta property="og:image" content="https://example.com/og/markup.png">

<meta name="twitter:card" content="summary_large_image">
Canonical, alternates, pagination
  • canonical for duplicate content (e.g., tracking params)
  • alternate for localized pages / language variants
  • prev/next only when paginated series is truly sequential 
<link rel="canonical" href="https://example.com/docs/markup">
<link rel="alternate" href="https://example.com/ja/docs/markup" hreflang="ja">
<link rel="alternate" href="https://example.com/docs/markup" hreflang="en">

Naming & Attributes

Naming must be predictable and searchable. Prefer kebab-case for HTML classes and data attributes.

Classes

  • Use BEM for components: .card, .card__title, .card--featured
  • Avoid deep descendant selectors; keep markup class hooks explicit
  • Do not use presentational names like .red or .mt-10 unless they are part of a documented utility system

IDs

  • Use IDs for document anchors (e.g., section navigation) and form relationships
  • Avoid IDs as styling hooks (hard to override and reuse)
  • IDs must be unique per page

Data attributes for JS

Prefer data-* hooks for JavaScript to decouple behavior from styling. 

<button class="btn btn--primary" type="button" data-modal-open="example">
  Open modal
</button>

<dialog class="modal" data-modal="example">...</dialog>

Links, Media, and Assets

Links must be explicit, accessible, and resilient. Media should be responsive and optimized.

Links
  • Use meaningful link text (avoid ‚Äúclick here‚Äù)
  • External links: target="_blank" must include rel="noopener"
  • Use root-relative paths for internal assets when possible 
<a href="/docs/markup.html">Read markup rules</a>
<a href="https://example.com" target="_blank" rel="noopener">External resource</a>
Images
  • Always provide alt. Use empty alt="" for decorative images
  • Prefer modern formats (WebP/AVIF) with fallbacks when necessary
  • Use width/height attributes to reduce layout shift
  • Use loading="lazy" for below-the-fold images 
<img
  src="./assets/images/pages/home/hero-main-960w.webp"
  srcset="
    /assets/images/pages/home/hero-main-640w.webp 640w,
    /assets/images/pages/home/hero-main-960w.webp 960w,
    /assets/images/pages/home/hero-main-1280w.webp 1280w
  "
  sizes="(max-width: 768px) 92vw, 960px"
  width="960"
  height="540"
  alt="Screenshot of the documentation layout"
  loading="lazy"
/>
Icons
  • Prefer inline SVG for UI icons when you need styling via CSS
  • Decorative icons must be hidden from screen readers (aria-hidden="true")
  • Do not use icon-only buttons without an accessible name 
<button class="icon-btn" type="button" aria-label="Open navigation">
  <svg aria-hidden="true" ...>...</svg>
</button>

Forms

Forms must be accessible and robust. Use correct input types, labels, and validation messaging.

Labels & relationships

  • Every input needs a programmatic label: <label for> or wrapping <label>
  • Group related controls with fieldset + legend
  • Use aria-describedby for hints and errors
<div class="form-field">
  <label class="form-field__label" for="email">Email</label>
  <input class="form-field__input" id="email" name="email" type="email" autocomplete="email" required />
  <p class="form-field__hint" id="email-hint">We will never share your email.</p>
</div>

Buttons

  • Use <button>, not <input type="submit">, for easier styling and richer content
  • Set explicit type on all buttons (button, submit, reset) 
<button class="btn btn--primary" type="submit">Send</button>
<button class="btn btn--secondary" type="button">Cancel</button>

‚ö†Ô∏è Validation

  • Use native HTML validation where appropriate (required, type, minlength)
  • Client-side validation must not replace server-side validation
  • Errors must be announced: use an error summary region and per-field messages

Accessibility Essentials

Accessibility is a product requirement. Markup must support keyboard, screen readers, and responsive zoom.

Topic Rule Example
Keyboard All interactive elements reachable and operable button over clickable div
Focus Visible focus indicator must exist :focus-visible styles
Names Controls have accessible names aria-label for icon-only buttons
Headings Logical heading order supports navigation h2 sections under one h1
Images Meaningful alt text or decorative alt="" Decorative SVG: aria-hidden="true"

Preferred patterns

  • Provide ‚ÄúSkip to content‚Äù link near the top
  • Use aria-current="page" for active nav links
  • Use native elements (details/summary, dialog) where possible

Markup for Performance

Markup choices directly impact loading speed and UX stability.

Reduce layout shift

High impact
  • Set image dimensions
  • Avoid injecting content above the fold
  • Reserve space for ads/embeds

Load JS responsibly

Important
  • Prefer type="module"
  • Defer non-critical scripts
  • Don‚Äôt block rendering with sync scripts

Fonts

Important
  • Preload primary WOFF2 fonts
  • Use font-display: swap
  • Limit font families (max 2‚Äì3)

Implementation Checklist

Before you ship

  • One h1 and correct heading hierarchy
  • Unique <title> and meta description per page
  • All form controls labeled; errors announced
  • Images have appropriate alt and dimensions
  • Navigation has aria-current and correct landmarks
  • Interactive UI works with keyboard only
  • No unnecessary wrapper divs; semantics are intentional

Tooling recommendations

  • Use Prettier for formatting consistency
  • Use HTML validation (W3C validator) for static HTML
  • Use ESLint rules to prevent anti-patterns (e.g., accessibility plugins)

Landmarks: Examples

Landmarks create a reliable page map for assistive technologies and improve keyboard navigation. Prefer native landmark elements over generic div wrappers.

Recommended structure 

<body>
  <a class="skip-link" href="#main-content">Skip to content</a>

  <header class="site-header">...</header>

  <nav class="nav-main" aria-label="Primary">...</nav>

  <main id="main-content">
    ...
  </main>

  <footer class="site-footer">...</footer>
</body>

Avoid

<div class="header">...</div>
<div class="nav">...</div>
<div class="main">...</div>

If a wrapper is required, keep it for layout only and still include the correct landmark element inside.

Practical checklist

  • One <main> per page
  • Navigation has an aria-label when multiple navs exist
  • ‚ÄúSkip to content‚Äù link points to the main container
  • Don‚Äôt nest landmarks unnecessarily (e.g., nav inside nav)

Headings: Rules

Headings are the primary navigation mechanism for screen readers. Maintain a consistent, logical hierarchy.

Good hierarchy 

<h1>Markup</h1>

<h2>Semantics</h2>
<h3>Landmarks</h3>

<h2>Forms</h2>
<h3>Labels</h3>

Common mistakes

  • Skipping levels (e.g., h2 ‚Üí h4)
  • Using headings only for styling (‚Äúbig text‚Äù) instead of meaning
  • Multiple h1 for unrelated page sections

Component rule

Reusable components should not hardcode heading levels unless the component is a page-level section. Prefer passing the heading level from the page context (or use a non-heading element with a class).

Sectioning: Patterns

Use section and article when they add meaning to the document outline. Otherwise, use div for purely visual wrappers.

When to use <article>

A self-contained item that can be reused or distributed independently (blog post, product card, news item). 

<article class="news-card">
  <h3>Performance audit results</h3>
  <p>...</p>
  <a href="/news/perf-audit.html">Read more</a>
</article>

When to use <section>

A thematic grouping on a page. It should almost always have a heading. 

<section aria-labelledby="pricing-title">
  <h2 id="pricing-title">Pricing</h2>
  ...
</section>

Avoid ‚Äúsection soup‚Äù 

<section><section><section>...</section></section></section>

Deep nesting usually indicates layout wrappers. Use div for layout and reserve sectioning elements for meaning.
